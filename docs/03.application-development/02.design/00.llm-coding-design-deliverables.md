# LLM 바이브 코딩용 설계산출물 정의

## 1. 목적

본 문서는 LLM을 활용하여 BMS를 구현할 때 필요한 설계산출물의 종류, 원본 문서, 작성 수준과 구현 착수 조건을 정의한다.

전통적인 SI 산출물을 모두 작성하는 것이 목적이 아니다. LLM이 기능의 목적과 범위를 이해하고, 화면·API·데이터·권한·예외처리 및 테스트를 일관되게 구현하는 데 필요한 최소한의 실행 가능한 설계 계약을 만드는 것이 목적이다.

---

## 2. 기본 원칙

### 2.1 설계는 구현 계약이다

설계 문서는 설명 자료가 아니라 다음 질문에 명확히 답해야 한다.

- 무엇을 구현하는가?
- 무엇을 구현하지 않는가?
- 누가 어떤 조건에서 사용하는가?
- 정상 흐름과 예외 흐름은 무엇인가?
- 화면과 API는 어떤 데이터를 주고받는가?
- 어떤 데이터와 제약조건을 사용하는가?
- 어떤 결과가 나오면 완료로 판단하는가?

### 2.2 하나의 사실은 하나의 원본에서 관리한다

- 요구 범위와 우선순위의 원본은 제품 백로그이다.
- 기능의 존재, 목적과 계층구조의 원본은 상세 업무기능 문서이다.
- 기능별 처리조건, 계산식, 상태전이, 등록·수정·삭제 제한 등 업무규칙의 원본은 기능 명세와 업무규칙 카탈로그이다.
- 테이블·컬럼·관계·제약조건의 원본은 데이터 카탈로그 CSV이다.
- API URL, Method, 요청·응답 스키마와 HTTP 상태코드의 원본은 OpenAPI 명세이다.
- API 처리순서, 트랜잭션, 동시성, 멱등성 및 감사로그 기준의 원본은 API 기능 명세이다.
- 화면 필드와 상호작용의 원본은 화면 상세설계이다.
- 완료 판단의 원본은 인수조건과 테스트케이스이다.
- 다른 문서에서는 원본을 복제하지 않고 식별자와 링크로 참조한다.

### 2.3 큰 문서보다 기능별 설계 묶음을 사용한다

LLM에는 시스템 전체 문서를 한꺼번에 제공하지 않는다. 구현할 기능에 필요한 문서만 묶은 `기능 설계 패키지`를 제공한다.

```text
기능 설계 패키지
├── manifest.yaml
├── 기능 명세
├── 화면 상세설계                      # 화면이 있는 경우
├── API 기능 명세와 OpenAPI 조각       # API가 있는 경우
├── 배치·외부연계 명세                 # 배치·연계가 있는 경우
├── 업무규칙·상태전이 참조             # 적용되는 경우
├── 데이터·조회모델 참조
├── 권한·예외 규칙
└── 인수조건·테스트케이스
```

### 2.4 식별자로 추적한다

모든 구현 단위는 가능한 경우 다음 식별자를 연결한다.

```text
백로그 ID
→ 요구사항 ID
→ BFD 기능 ID
→ 화면 ID
→ API ID 또는 배치·연계 ID
→ 테이블
→ 테스트케이스 ID
```

## 3. 설계산출물 체계

### 3.1 레벨 1: 개발 기준선

프로젝트 전체에 공통으로 적용되며 자주 변경하지 않는 문서이다.

| 산출물 | 목적 | 필수 여부 | 현재 원본 |
| --- | --- | :---: | --- |
| 제품 백로그·릴리스 범위 | 버전별 포함·제외 범위와 우선순위 확정 | 필수 | `00.project-management/product-backlog.md` |
| 기능 카탈로그 | BFD L4 기능별 화면·API·엔터티·설계 및 구현 상태 관리 | 필수 | 작성 필요 |
| 애플리케이션 아키텍처 | 모듈 경계, 계층, 기술 스택과 호출 원칙 정의 | 필수 | `02.architecture/01.application-architecture/application-architecture.md` |
| 데이터 아키텍처 | 표준, 논리·물리 모델, 제약조건과 ERD 정의 | 필수 | `02.architecture/02.data-architecture/` |
| 용어·코드 기준 | 도메인 용어와 상태·유형 코드 의미 통일 | 필수 | 데이터 표준 및 상세 업무기능 문서 |
| 구현 규약 | 패키지, 예외, 로깅, 트랜잭션, 테스트 구조 정의 | 필수 | 작성 또는 정비 필요 |
| 설계 결정 기록 ADR | 중요한 선택의 배경, 결정과 영향 보존 | 결정 발생 시 필수 | 데이터 아키텍처 ADR 존재, 애플리케이션 ADR 작성 필요 |

제품 백로그의 기능 항목은 최소한 백로그 ID, 업무영역, 기능명, 릴리스, 우선순위, 포함 범위, 제외 범위, 요구사항 ID, BFD ID와 상태를 식별할 수 있어야 한다. 릴리스 공통 범위는 로드맵에서 한 번 정의하고 기능별 예외만 개별 항목에 기록한다.

### 3.2 레벨 2: 공통 인터페이스 계약

모든 기능에서 반복 적용하는 기술·사용자 경험 규칙이다.

| 산출물 | 필수 내용 | 필수 여부 |
| --- | --- | :---: |
| API 공통 규칙 | URL, HTTP Method, 상태코드, 응답 형식, 페이징, 정렬, 검색조건 | 필수 |
| OpenAPI 기본 명세 | 공통 스키마·보안 정의와 API 계약 조각 통합 기준 | 필수 |
| 오류코드 정의 | 오류코드, 사용자 메시지, HTTP 상태, 로그 수준 | 필수 |
| 인증·권한 설계 | 로그인, 세션, CSRF, RBAC, 기능권한과 필요 시 행 단위 권한 | 필수 |
| 권한 매트릭스 | 역할·기능·API·메뉴 접근권한 연결 | 필수 |
| 화면 공통 규칙 | 목록·상세·폼, 검증 메시지, 로딩·빈 상태·오류 상태 | 필수 |
| 감사·로그 규칙 | 접속로그, 시스템로그, 요청추적ID, 개인정보 마스킹 | 필수 |
| 개인정보 처리 기준 | 개인정보 분류, 마스킹, 로그·다운로드 허용 및 보존 기준 | 필수 |
| DB 마이그레이션 규칙 | 도구, 파일명, 불변성, 초기 데이터 및 카탈로그 동기화 | 필수 |
| 파일 처리 규칙 | 업로드, 다운로드, 크기·형식 검증, 저장 실패 보상 | 해당 시 필수 |
| 배치·외부연계 규칙 | 실행, 재시도, 멱등성, 잠금, 오류·재처리 | 해당 시 필수 |

### 3.3 레벨 3: 기능 설계 패키지

하나의 사용자 가치 또는 유스케이스를 구현하기 위한 핵심 산출물이다.

#### 기능 명세

- 백로그 ID와 BFD 기능 ID
- 목적과 사용자
- 포함 범위와 제외 범위
- 선행조건
- 정상 처리 흐름
- 상태 변경
- 업무 규칙
- 예외 및 오류처리
- 권한
- 후속 처리와 연계 기능

상세 업무기능 문서는 기능의 존재와 계층을 식별하는 분석 원본으로 사용한다. 실제 처리 규칙은 기능 명세에 작성하고, 여러 기능에서 공유하는 규칙은 업무규칙 카탈로그의 `rule_id`를 참조한다. 상태를 가진 엔터티를 변경하는 기능은 상태전이 카탈로그의 허용 전이를 참조한다.

#### 화면 상세설계

화면이 있는 기능에 작성한다.

- 화면 ID와 경로
- 화면 목적과 접근권한
- 조회조건, 목록 컬럼과 정렬
- 입력 필드, 타입, 필수성, 기본값과 검증
- 버튼과 사용자 동작
- 모달·탭·페이지 이동
- 로딩·빈 상태·오류·권한 없음 상태
- 호출 API ID
- 반응형 또는 접근성 요구사항
- 렌더링 경계와 Server/Client Component 구분
- URL Query Parameter와 화면상태 보존 기준
- 폼 검증 스키마
- 저장 후 이동, 캐시 무효화 및 재조회 기준

프론트엔드 기술 항목은 구현 방식에 영향을 주는 경우에만 명시한다. 와이어프레임은 복잡한 배치나 상호작용이 있을 때만 작성한다.

#### OpenAPI 명세

기계 판독 가능한 API 계약 원본으로 관리한다.

- API ID 또는 Operation ID
- Method와 URL
- 인증·보안 요구사항
- Path, Query와 Header Parameter
- Request와 Response Schema
- 필드 타입, 필수성, 형식과 허용값
- HTTP 상태코드와 공통 오류 응답
- 요청·응답 예시

OpenAPI는 Spring Controller, TypeScript Client, Mock Server 및 계약 테스트 자동화의 기준으로 사용한다.

#### API 기능 명세

- API ID, 기능 ID와 화면 ID
- 인증·권한
- 업무 처리 순서와 적용 업무규칙 ID
- OpenAPI Operation ID와 업무 오류코드
- 트랜잭션 경계
- 멱등성·동시성 처리
- 참조 테이블과 주요 제약조건
- 감사이력과 변경 전후값 기록 여부
- 후속 이벤트·알림과 다른 업무영역 영향
- 캐시 무효화 대상

#### 배치·외부연계 상세명세

해당 기능에만 작성한다.

- 프로그램 ID와 실행 주체
- 실행 주기 또는 호출 조건
- 입력·출력과 외부 시스템
- 처리 순서
- 페이지 처리와 호출량 제한
- 타임아웃·재시도·멱등성
- 실행 잠금
- 부분실패와 재처리
- 로그·모니터링·알림

#### 데이터 참조

테이블 정의를 기능 문서에 복제하지 않는다. 다음 내용만 연결한다.

- 조회·생성·수정·삭제 대상 테이블
- 주요 PK·FK·UK·CHECK
- 트랜잭션에서 함께 변경하는 엔터티
- 논리삭제와 이력 보존 기준
- 애플리케이션에서 집행하는 참조·정합성 규칙

#### 조회·집계모델 정의

단순 엔터티 CRUD가 아닌 조회·집계 기능에 조건부로 작성한다.

- 조회모델 ID와 목적
- 결과 필드와 타입
- 원천 엔터티·테이블
- 조인, 필터와 집계 산식
- 기준 시점과 상태·논리삭제 포함 기준
- 권한 범위
- 관련 API ID

대시보드, 인력 가용현황, 고객 거래이력, 프로젝트 종합현황 및 손익처럼 여러 원천 데이터를 조합하는 기능은 조회·집계모델 정의 없이 구현하지 않는다.

#### 인수조건과 테스트케이스

- 정상 시나리오
- 필수값·경계값
- 권한 없음
- 존재하지 않거나 삭제된 데이터
- 상태 전이 오류
- 중복·동시 요청
- 트랜잭션 롤백
- 개인정보·로그 마스킹
- API 계약 및 화면 동작

인수조건은 가능한 경우 `Given / When / Then` 형식으로 작성한다.

#### 기능 패키지 Manifest

LLM이 읽을 문서와 변경 가능한 코드 범위를 기계적으로 제한하는 `manifest.yaml`을 기능 패키지마다 작성한다.

```yaml
package_id: FPK-COM-LOGIN-001
title: 로그인
backlog_id: PB-001
function_ids:
  - BFD-02-05-01-01
references:
  screens:
    - screens/SCR-COM-LOGIN-001.md
  apis:
    - apis/API-COM-AUTH-001.md
  openapi:
    - ../../04.openapi/paths/auth.yaml
  rules:
    - BR-AUTH-001
  tests:
    - tests/acceptance.feature
allowed_paths:
  - backend/src/main/java/com/bms/backend/common/**
  - frontend/app/(auth)/login/**
  - frontend/features/auth/**
forbidden_paths:
  - backend/src/main/java/com/bms/backend/customer/**
```

Manifest에는 최소한 패키지 ID, 기능 ID, 참조 문서, 허용 변경 경로와 변경 금지 경로를 둔다. DB 변경이 있으면 마이그레이션 경로와 데이터 카탈로그 참조를 추가한다.

### 3.4 구조화 카탈로그

카탈로그는 전체 설계 상태와 참조 무결성을 관리하는 원본이며 Markdown 설명을 대체하지 않는다.

#### 기능 카탈로그

파일은 `02.design/02.catalog/functions.csv`로 관리한다.

```text
function_id,function_name,domain,requirement_id,backlog_id,function_type,screen_ids,api_ids,primary_entities,design_status,implementation_status
```

- 모든 상세 BFD L4 기능을 한 건씩 등록한다.
- `function_type`은 `COMMAND`, `QUERY`, `BATCH`, `INTERFACE` 등 통제된 값을 사용한다.
- 복수 식별자는 정해진 구분자로 기록하고 검증기가 실제 카탈로그 등록 여부를 검사한다.
- 설계·구현 상태의 허용값과 전이 기준은 카탈로그 스키마에서 별도로 정의한다.

#### 업무규칙 카탈로그

파일은 `02.design/02.catalog/business-rules.csv`로 관리한다.

```text
rule_id,domain,rule_name,rule_type,condition,result,error_code,related_entities
```

- 여러 기능·화면·API에서 공유하거나 데이터 제약만으로 표현할 수 없는 규칙을 등록한다.
- 기능 하나에서만 사용하는 단순 처리 순서는 기능 명세에 두며 카탈로그를 불필요하게 세분화하지 않는다.
- 기능 명세와 API 기능 명세에서는 규칙 문장을 복제하지 않고 `rule_id`를 참조한다.

#### 상태전이 카탈로그

파일은 `02.design/02.catalog/state-transitions.csv`로 관리한다.

```text
transition_id,entity,current_status,event,next_status,condition_rule_id,permission_code
```

- 상태를 가진 엔터티의 상태 변경 기능에는 필수다.
- 허용 전이뿐 아니라 변경 권한과 조건 업무규칙을 연결한다.
- 상태 다이어그램은 복잡한 전이를 시각화할 때만 추가하는 선택 산출물이다.

#### 조회모델 카탈로그

파일은 `02.design/02.catalog/view-models.csv`로 관리한다.

```text
view_model_id,view_model_name,domain,source_entities,related_api_ids,detail_spec
```

- 여러 엔터티를 조합하거나 계산·집계하는 조회 기능을 등록한다.
- 필드별 산식과 포함 기준은 `detail_spec`이 가리키는 조회·집계모델 상세문서에서 관리한다.

#### 기타 참조 카탈로그

화면, API, 권한, 오류코드 및 추적성은 다음 구조화 카탈로그에서 관리한다.

- `screens.csv`
- `apis.csv`
- `permissions.csv`
- `error-codes.csv`
- `traceability.csv`

### 3.5 레벨 4: 구현 작업카드

LLM에 실제 코딩을 지시할 때 제공하는 가장 작은 실행 단위이다.

| 항목 | 내용 |
| --- | --- |
| 작업 목표 | 이번 작업에서 완료할 사용자 가치 |
| 참조 설계 | 기능·화면·API·데이터·테스트 문서 링크 |
| 수정 범위 | 백엔드·프론트엔드·DB의 대상 패키지와 파일 |
| 제외 범위 | 이번 작업에서 변경하지 않을 기능 |
| 구현 순서 | 마이그레이션, 백엔드, 프론트엔드, 테스트 등 |
| 완료 조건 | 실행할 테스트와 확인할 결과 |
| 주의사항 | 보안, 개인정보, 동시성, 기존 호환성 |
| 기능 패키지 | 기능 설계 패키지 ID와 Manifest 경로 |
| 변경 경계 | 허용 경로, 금지 경로와 생성·수정 예상 파일 |
| DB 변경 | 마이그레이션 필요 여부와 카탈로그 선변경 여부 |
| 검증 명령 | 백엔드·프론트엔드·계약·카탈로그 검증 명령 |
| 후속 문서 | 구현 후 갱신할 상태·추적성 문서와 ADR 필요 여부 |

작업카드는 장기 원본 문서가 아니라 설계 원본을 조합한 실행 지시서이다. 구현 완료 후 코드와 테스트가 결과물이 되며, 새로운 설계 결정이 생긴 경우에만 원본 설계 문서를 갱신한다.

검증 명령은 “테스트 수행”처럼 추상적으로 쓰지 않고 저장소에서 실행 가능한 명령을 기록한다. 예를 들어 Maven 테스트, 프론트엔드 lint·typecheck·test·build 및 데이터 카탈로그 검증 명령을 프로젝트의 실제 스크립트에 맞춰 명시한다.

### 3.6 DB 변경 및 마이그레이션 계약

첫 스키마 구현 전에 마이그레이션 도구와 운영 규칙을 ADR로 확정한다. 도구를 확정한 뒤에는 다음 원칙을 적용한다.

- 테이블·컬럼·제약조건 변경은 데이터 카탈로그를 먼저 변경하고 검증한다.
- 카탈로그 변경에 대응하는 버전 마이그레이션을 추가한다.
- 이미 공유 환경에 적용된 마이그레이션은 수정하거나 순번을 재사용하지 않는다.
- 마이그레이션 파일명, 버전 부여와 반복 가능 마이그레이션 사용 기준을 통일한다.
- 공통코드·시스템설정 등 운영 기준정보와 개발·테스트 샘플 데이터를 분리한다.
- 스키마 초기화, 업그레이드와 롤백 또는 보상 절차를 테스트한다.
- 실제 DB 스키마, 마이그레이션과 데이터 카탈로그의 불일치를 자동 검증한다.

### 3.7 설계 결정 기록

여러 구현 방식이 가능하고 이후 기능에 지속적인 영향을 주는 결정은 ADR로 기록한다.

- 상태: Proposed, Accepted, Superseded 또는 Rejected
- 배경과 해결할 문제
- 검토한 대안과 장단점
- 선택한 결정과 근거
- 데이터·API·보안·운영·테스트 영향
- 대체하거나 후속 검토할 조건

단순 구현 세부사항은 ADR을 만들지 않는다. 인증 방식, 행 단위 권한 적용 여부, API 응답 래퍼, DB 마이그레이션 도구, 파일 저장소 및 동시성 전략처럼 여러 기능에 영향을 주는 결정에 사용한다.

### 3.8 개인정보 처리 기준

개인정보 카탈로그에는 다음 정보를 구조화하여 관리한다.

```text
data_item,entity,sensitivity,masking_rule,log_allowed,download_allowed,retention_policy
```

- 사용자·고객담당자·직원·외주인력의 연락처와 이메일
- 생년월일과 주소
- 로그인ID와 접속 IP 등 인증·보안 정보
- 외부연계 응답에 포함될 수 있는 개인 식별정보

필드별 조회권한, 화면 마스킹, 로그 기록 허용, 다운로드 권한과 보존·파기 기준을 확정한다. 단순히 “개인정보를 보호한다”는 문구만으로 구현 준비 완료로 판단하지 않는다.

### 3.9 설계 자동 검증

구조화 카탈로그와 기능 패키지는 다음 항목을 자동 검증한다.

- 모든 BFD L4 기능이 기능 카탈로그에 등록되어 있는가
- 화면·API·테스트의 기능 ID가 실제 BFD 기능을 참조하는가
- 화면이 참조하는 API ID와 API가 참조하는 화면 ID가 등록되어 있는가
- OpenAPI Operation ID와 API 카탈로그가 일치하는가
- API 요청·응답 스키마가 유효하고 OpenAPI 문법 검증을 통과하는가
- 참조 엔터티·테이블·컬럼이 데이터 카탈로그에 존재하는가
- 업무규칙, 권한, 오류코드와 상태코드 ID가 등록되어 있는가
- 상태전이의 현재·다음 상태와 권한이 유효한가
- 테스트케이스가 기능 ID와 인수조건을 참조하는가
- 식별자가 중복되거나 삭제된 식별자가 재사용되지 않았는가
- Manifest의 참조 파일과 허용·금지 경로가 유효한가

자동 검증은 문서 내용의 업무적 타당성을 대신하지 않는다. 참조 무결성과 형식 오류를 조기에 발견하여 사람이 업무 결정과 예외 검토에 집중하도록 한다.

## 5. LLM 구현 준비 완료 기준

기능은 다음 조건을 모두 만족할 때 구현 준비 완료(`Definition of Ready`)로 판단한다.

- [ ] 백로그 버전과 포함·제외 범위가 확정되어 있다.
- [ ] BFD L4 기능이 기능 카탈로그에 등록되어 있고 설계 상태가 구현 가능 상태이다.
- [ ] 기능 명세와 적용 업무규칙 ID가 정의되어 있다.
- [ ] 사용자와 필요한 권한이 정의되어 있다.
- [ ] 정상 흐름과 주요 예외 흐름이 정의되어 있다.
- [ ] 상태를 변경하는 기능이면 허용 상태전이와 조건이 정의되어 있다.
- [ ] 화면이 있으면 필드·동작·상태와 호출 API가 정의되어 있다.
- [ ] API가 있으면 OpenAPI 요청·응답 계약과 API 기능 명세가 정의되어 있다.
- [ ] 배치·연계가 있으면 재시도·멱등성·잠금·재처리가 정의되어 있다.
- [ ] 사용하는 테이블과 주요 제약조건을 식별할 수 있다.
- [ ] 조회·집계 기능이면 결과 필드, 원천 데이터와 산식이 정의되어 있다.
- [ ] 트랜잭션 경계와 동시성 규칙이 정의되어 있다.
- [ ] 개인정보 분류와 화면·로그·다운로드·보존 처리 기준이 정의되어 있다.
- [ ] 인수조건과 테스트 시나리오가 정의되어 있다.
- [ ] Manifest에 참조 문서와 허용·금지 변경 경로가 정의되어 있다.
- [ ] 구현 작업카드에 실행 가능한 검증 명령이 정의되어 있다.
- [ ] 관련 설계 문서 사이에 알려진 모순이 없다.
- [ ] 설계 카탈로그와 OpenAPI 자동 검증을 통과한다.

하나라도 구현 결과를 크게 바꿀 수 있는 항목이 미확정이면 LLM 코딩 전에 결정한다. 단순한 표현이나 내부 구현 선택은 작업카드에서 합리적인 기본값을 허용할 수 있다.

## 8. 권장 진행 순서

1. v1.0 최소 프로젝트 범위를 애플리케이션 아키텍처와 데이터 모델에 반영한다.
2. 기존 패키지 구조 문서와 현재 애플리케이션 아키텍처의 모듈 명칭을 통일한다.
3. 기능 카탈로그를 만들고 1차 BFD L4 기능의 범위와 설계 상태를 등록한다.
4. API·OpenAPI·오류·권한·화면·로그·개인정보·DB 마이그레이션·테스트 공통 규칙을 확정한다.
5. 인증·사용자·권한을 첫 번째 기능 설계 패키지 묶음으로 작성한다.
6. 로그인, 현재 사용자 조회, 사용자 목록·등록, 역할 부여, 메뉴권한 조회와 로그아웃을 첫 번째 수직 단위로 구현한다.
7. 고객사·고객담당자, 영업기회, 수동 사업공고, 최소 프로젝트, 인력 순으로 기능 설계 패키지와 구현을 반복한다.
8. 사업공고 자동수집처럼 복잡한 기능은 별도 배치·외부연계 상세명세와 시퀀스 다이어그램을 작성한 후 구현한다.

첫 번째 수직 단위 전에 사용자–직원 관계, 사용자 다중 역할, 역할–메뉴 권한, 인증 방식, 논리삭제, API 응답 형식과 DB 마이그레이션 방식을 기준선 또는 ADR로 확정한다. 로그인만 고립하여 구현하지 않고 사용자·역할·메뉴 구조가 실제로 동작하는 최소 범위를 함께 검증한다.

## 9. 완료 정의

설계 단계는 문서 개수가 아니라 다음 상태를 만족하면 완료된 것으로 본다.

- 구현 대상 기능이 기능 설계 패키지로 준비되어 있다.
- 기능 Manifest만으로 필요한 컨텍스트와 허용 변경 범위를 기계적으로 구성할 수 있다.
- LLM이 추가적인 업무 추측 없이 화면·API·DB·테스트를 구현할 수 있다.
- 구현 결과를 판단할 자동화 테스트와 인수조건이 존재한다.
- 기능·화면·API·규칙·상태·데이터·테스트 간 참조가 자동 검증된다.
- 설계 원본 간 중복과 모순이 통제된다.
- 변경된 결정이 관련 원본 문서와 테스트에 반영된다.
