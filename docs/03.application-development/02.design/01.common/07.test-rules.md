# 테스트 공통 규칙

## 1. 테스트 수준과 채택 도구

| 수준 | 대상 | 채택 도구 |
| --- | --- | --- |
| 단위 | 도메인 규칙, 상태 전이, 값 검증, 권한 판정 | JUnit 5, AssertJ, Mockito |
| 저장소·마이그레이션 | JPA 매핑, PostgreSQL 제약, Flyway 전체 적용 | Spring Boot Test, Testcontainers PostgreSQL |
| API 통합 | 인증·CSRF·권한, 요청 검증, 오류·페이징 계약 | MockMvc와 실제 PostgreSQL 컨테이너 |
| 프론트엔드 | 컴포넌트, 폼, 공통 상태와 권한 표시 | Vitest, React Testing Library |
| E2E | 사용자 관점 핵심 흐름 | Playwright |
| 계약 | OpenAPI 문법, operationId, 요청·응답 호환성 | OpenAPI lint와 계약 테스트 |

현재 프로젝트에 없는 테스트 의존성과 script는 해당 수준의 첫 기능 구현 시 추가한다.

## 2. 필수 통과 기준

- 변경된 업무 규칙에는 정상·경계·거절 사례가 있어야 한다.
- 모든 보호 API는 미인증 401, 무권한 403, 허용 2xx 사례를 검증한다.
- 상태 변경은 허용 전이, 금지 전이, 동시 수정과 감사로그를 검증한다.
- 개인정보 마스킹, 원문 조회 권한, 다운로드 차단과 로그 비노출을 검증한다.
- 모든 마이그레이션은 빈 PostgreSQL에 처음부터 적용되고 JPA 검증을 통과해야 한다.
- OpenAPI 명세와 실제 API의 상태코드·오류 구조가 일치해야 한다.
- 결함 수정에는 재현 테스트를 먼저 추가한다.

초기 품질 게이트는 전체 라인 커버리지 70%, 변경 코드 80%를 목표로 하되 수치만으로 승인하지 않는다. 인증, 권한, 개인정보, 프로젝트 기간·상태, 인력투입과 원가 규칙의 필수 시나리오가 하나라도 실패하면 병합하지 않는다.

## 3. 테스트 데이터

- 운영 개인정보와 운영 DB 덤프를 개발·테스트에 사용하지 않는다.
- 고정된 가상 사용자와 프로젝트를 builder 또는 fixture로 만들고 테스트 간 상태를 공유하지 않는다.
- 시간대 테스트는 UTC 저장과 Asia/Seoul 표시, 자정·월말·윤년 경계를 포함한다.
- 비밀번호, 토큰과 비밀키는 테스트 전용 값을 사용하고 실패 출력에도 원문이 나타나지 않게 한다.

## 4. 시간·동시성 테스트

- 세션·토큰·유효기간 구현은 주입 가능한 `Clock`을 사용하고 테스트에서 실제 시간을 기다리지 않는다.
- 시간 경계는 만료 직전, 정확한 만료시각과 만료 직후를 검증하며 정확한 만료시각부터 만료로 판정한다.
- 동시성·DB 제약 테스트는 실제 PostgreSQL Testcontainers와 서로 다른 트랜잭션을 사용한다.
- 메모리 저장소 모의 테스트만으로 로그인 실패건수, 중복 등록과 보안버전 동시성 검증을 대체하지 않는다.

## 5. 보안 테스트

- 비밀번호 해시 알고리즘·파라미터, 세션 고정 방지, 쿠키, CSRF, 유휴·절대 만료와 재인증 경계를 검증한다.
- 역할·메뉴권한, 계정상태와 보안버전 변경 후 기존 세션 차단을 검증한다.
- 오류응답과 로그에 비밀번호·해시·세션ID·쿠키·CSRF·Authorization 헤더와 내부 예외가 포함되지 않는지 검사한다.
- 운영 배포 전 HTTPS 전환, HSTS, CSP, 캐시 금지와 신뢰 프록시 설정을 환경 수준에서 확인한다.

## 6. 시나리오 검증

기능 패키지 Manifest에 선언한 Gherkin 파일, 시나리오 구조와 API·외부 오류코드 참조는 다음 명령으로 검증한다.

```text
python scripts/validate_test_scenarios.py
```

선언되지 않은 `.feature`, 중복 시나리오명, `만일`·`그러면` 단계 누락과 미등록 API·오류코드는 구현 전에 수정한다.
