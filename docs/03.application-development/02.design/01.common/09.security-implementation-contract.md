# 보안 구현 계약

## 1. 목적과 적용 원칙

이 문서는 인증·인가 기능을 구현·검토·시험할 때 반드시 지켜야 하는 기술 계약이다. 요구사항이 명시되지 않은 보안 동작은 허용으로 추정하지 않고 기본 거부한다.

- `MUST`: 구현과 배포 전에 반드시 충족한다.
- `MUST NOT`: 예외 없이 금지한다.
- `SHOULD`: 적용하지 않으려면 위험과 대안을 ADR로 승인받는다.

인증 정책의 업무 원본은 `01.authentication-authorization-design.md`, API 오류 형식은 `02.api-openapi-error-rules.md`, 초기 데이터와 최초 관리자 생성은 `08.initial-data-bootstrap-design.md`를 따른다.

## 2. 통신과 배포 경계

- 운영의 모든 화면과 API는 HTTPS만 사용해야 한다.
- HTTP 요청은 HTTPS로 전환하고 HSTS를 적용해야 한다.
- 프론트엔드와 API는 동일 출처로 제공하며 CORS는 기본 비활성화해야 한다.
- 별도 출처가 필요하면 정확한 허용 출처, Method, Header와 자격 증명 전송 여부를 명시하여 재승인해야 한다. 와일드카드 출처와 자격 증명을 함께 허용해서는 안 된다.
- 클라이언트 IP와 HTTPS 여부를 판단하는 전달 헤더는 등록된 리버스 프록시에서 온 요청만 신뢰해야 한다.
- 운영 필수 비밀값과 보안 설정이 없거나 개발용 기본값이면 애플리케이션 기동을 실패시켜야 한다.

## 3. 비밀번호 처리

- 비밀번호는 Argon2id로 단방향 해시해야 한다.
- 최소 파라미터는 메모리 19 MiB, 반복 2회, 병렬도 1이며 운영 장비에서 성능을 측정한 뒤 동등 이상의 비용으로 조정해야 한다.
- 사용자와 비밀번호 변경 건마다 새로운 임의 salt를 사용해야 한다.
- 알고리즘과 파라미터를 포함한 인코딩 문자열을 저장하여 로그인 성공 시 더 강한 설정으로 재해시할 수 있어야 한다.
- 원문 비밀번호는 저장·캐시·세션 보관·로그 기록·응답 반환해서는 안 된다.
- 비밀번호 입력에는 공백 제거, 대소문자 변환과 유니코드 정규화를 적용해서는 안 된다.
- 로그인에서는 입력 크기 상한만 확인하고 신규 비밀번호 복잡도 실패 여부를 노출해서는 안 된다.
- 존재하지 않는 로그인ID에도 고정된 더미 해시를 검증하여 계정 존재 여부에 따른 응답시간 차이를 줄여야 한다.
- v1.0에서는 pepper를 사용하지 않는다. 비밀관리 시스템과 키 교체 절차를 갖춘 뒤 별도 승인한다.

## 4. 세션과 쿠키

| 항목 | 구현 계약 |
|---|---|
| 쿠키명 | `BMSSESSION` |
| `HttpOnly` | 항상 설정 |
| `Secure` | 운영환경 필수 |
| `SameSite` | `Lax` |
| `Path` | `/` |
| `Domain` | 지정하지 않음 |
| `Max-Age`·`Expires` | 지정하지 않는 브라우저 세션 쿠키 |
| URL 세션 식별자 | 금지 |

- 로그인 성공, 최초 등록 완료와 재인증 성공 시 세션 고정 공격 방지 전략을 적용해야 한다. 로그인과 최초 등록 완료는 세션ID를 교체하고, 재인증은 현재 세션을 유지하면서 재인증시각만 갱신한다.
- 로그아웃은 접속로그 처리 후 서버 세션을 무효화하고 쿠키를 만료시켜야 한다.
- 일반 세션은 사용자 활동 기준 유휴 15분과 로그인 성공 기준 절대 8시간을 함께 적용해야 한다.
- 제한 세션은 유휴 10분, 절대 30분, 임시 비밀번호 만료시각 중 가장 빠른 조건으로 만료해야 한다.
- 세션에는 사용자 식별정보, 역할·메뉴 표시 스냅샷, `securityVersion`, `lastUserActivityAt`, `absoluteExpiresAt`, `reauthenticatedAt`과 접속로그ID만 필요한 범위로 저장해야 한다.
- 영속성 모델, 비밀번호·해시, 세션ID 원문, 쿠키값과 CSRF 토큰을 세션 업무 DTO에 저장해서는 안 된다.

## 5. 세션 유효성 검증

모든 보호 API 요청은 업무 처리를 시작하기 전에 다음 순서로 검증해야 한다.

1. 서버 세션 존재 여부를 확인한다.
2. 유휴 만료와 절대 만료를 확인한다.
3. 현재 사용자의 활성·삭제 상태를 DB에서 확인한다.
4. DB의 `SEC_VER`와 세션의 `securityVersion`이 같은지 확인한다.
5. 제한 세션이면 허용 API와 자격 증명 기준시각을 추가 확인한다.
6. API의 메뉴·기능 권한과 데이터 범위 권한을 확인한다.

하나라도 실패하면 업무 서비스를 호출하지 않고 세션을 만료한다. 외부에는 `401 AUTH_AUTHENTICATION_REQUIRED`를 반환하고 내부 로그에 만료 원인을 구분한다.

다음 변경은 대상 사용자의 `SEC_VER`를 원자적으로 1 증가시켜야 한다.

- 사용자 역할 부여·회수
- 사용자가 가진 역할의 메뉴권한 추가·회수
- 사용자가 가진 역할의 논리 삭제
- 계정 활성·비활성·삭제
- 관리자 비밀번호 초기화
- 최초 등록 완료와 사용자 비밀번호 변경
- 보안상 기존 세션 폐기가 필요한 후속 정책 변경

역할의 메뉴권한을 변경할 때는 그 역할을 가진 모든 사용자의 보안버전을 같은 트랜잭션에서 증가시킨다. 세션 스냅샷을 직접 수정하지 않는다.

## 6. CSRF

- Spring Security의 CSRF 보호를 비활성화해서는 안 된다.
- `POST`, `PUT`, `PATCH`, `DELETE`와 상태를 변경하는 모든 요청은 CSRF 토큰을 검증해야 한다.
- 로그인, 로그아웃, 활동 갱신과 재인증도 CSRF 검증 대상이다.
- 클라이언트는 `GET /api/v1/auth/csrf`로 토큰을 받고 `X-CSRF-TOKEN` 헤더로 전송해야 한다.
- 로그인 성공, 로그아웃과 최초 등록 완료 후 이전 토큰을 폐기하고 클라이언트가 새 토큰을 조회하게 해야 한다.
- `SameSite=Lax`와 Origin 검사는 보조 통제이며 CSRF 토큰을 대체하지 않는다.
- 실패 응답은 `403 AUTH_CSRF_INVALID`를 사용하고 토큰값이나 검증 상세를 노출해서는 안 된다.

## 7. 인가와 중요 작업 재인증

- 공개 API는 정확한 허용목록으로 관리하고 나머지 요청은 기본적으로 인증을 요구해야 한다.
- 새 보호 API에 메뉴·기능 권한이 연결되지 않았으면 접근을 거부해야 한다.
- 프론트엔드 메뉴 숨김과 버튼 제어를 서버 인가 대신 사용해서는 안 된다.
- 서버는 화면·API 권한 이후 대상 데이터의 조직·담당자·본인 등 데이터 범위를 검사해야 한다.
- 시스템관리자도 명시적인 RBAC 관계를 사용하며 역할명이나 계정으로 우회해서는 안 된다.

다음 중요 작업은 로그인 또는 최근 재인증 성공 후 10분 안에서만 허용한다.

- 시스템관리자 역할 부여·회수
- 역할의 메뉴권한 변경과 역할 삭제
- 다른 사용자 비밀번호 초기화
- 관리자의 사용자 활성·비활성 처리
- 보안 설정과 감사로그 내보내기

재인증은 `POST /api/v1/auth/reauthenticate`에서 현재 비밀번호와 CSRF 토큰을 검증한다. 성공하면 세션의 `reauthenticatedAt`을 현재시각으로 갱신한다. 비밀번호 불일치는 `403 AUTH_REAUTHENTICATION_FAILED`를 반환하되 계정 로그인실패건수를 변경하지 않으며, 재인증 요청은 세션별 분당 5회로 제한하고 보안로그를 생성한다. 중요 작업의 재인증이 없거나 10분이 지났으면 `403 AUTH_REAUTHENTICATION_REQUIRED`를 반환한다.

## 8. 입력·오류·로그

- 모든 입력은 형식과 최대 길이를 서버에서 검증하고 SQL은 파라미터 바인딩을 사용해야 한다.
- 클라이언트 정렬값과 검색식을 SQL 조각에 직접 연결해서는 안 된다.
- 오류응답에는 예외 클래스, 스택 추적, SQL, 테이블명, 계정 존재 여부와 인증정보를 포함해서는 안 된다.
- 로그인 성공·실패·속도 제한, 세션 만료, CSRF 실패, 권한 거부, 재인증, 계정·역할·권한·비밀번호 변경과 부트스트랩 결과를 보안로그로 남겨야 한다.
- 로그에는 비밀번호·해시, 세션ID·쿠키, CSRF 토큰, Authorization 헤더, 암호화 키와 전체 요청·응답 본문을 기록해서는 안 된다.
- 모든 오류와 보안 이벤트는 인증비밀이 아닌 요청추적ID로 연결해야 한다.

## 9. 브라우저 보안 헤더와 캐시

- 운영 응답에 HSTS, `X-Content-Type-Options: nosniff`, `Referrer-Policy: same-origin`과 불필요한 기능을 제한하는 `Permissions-Policy`를 적용해야 한다.
- CSP는 nonce 기반을 기본으로 하고 최소한 `object-src 'none'`, `base-uri 'self'`, `frame-ancestors 'none'`을 포함해야 한다.
- 로그인, 현재 사용자, CSRF와 재인증 응답에는 `Cache-Control: no-store`를 적용해야 한다.
- 프레임 내 표시가 승인되지 않은 화면은 CSP `frame-ancestors`로 차단해야 한다.

## 10. 비밀값과 의존성

- DB 자격 증명, 로그인ID 비식별 키와 배포 비밀은 소스·CSV·Flyway·이미지에 넣지 않고 환경별 비밀 저장소에서 주입해야 한다.
- 비밀값을 명령행 인자로 전달하거나 로그에 출력해서는 안 된다.
- 운영 런타임 DB 계정은 애플리케이션 DML에 필요한 최소 권한만 가져야 한다.
- 의존성 잠금파일을 유지하고 CI에서 알려진 취약점과 비밀값 커밋을 검사해야 한다.
- 보안 라이브러리 버전과 지원 종료 여부는 배포 전에 확인해야 한다.

## 11. MFA 범위

v1.0 인증 방식은 로그인ID·비밀번호와 서버 세션으로 유지한다. 시스템관리자 MFA는 v1.0 이후 우선순위가 높은 보안 백로그로 등록하며 다음 중 하나를 별도 ADR에서 선택한다.

- 조직의 외부 IdP와 OIDC 연동
- WebAuthn 패스키 또는 보안키
- TOTP와 복구코드

MFA 도입 전까지 시스템관리자 계정에는 전체 메뉴 명시 권한, 8시간 절대 세션, 10분 중요 작업 재인증, 로그인 속도 제한과 보안로그를 적용한다.

## 12. 필수 보안 시험

- 비밀번호 해시 형식·파라미터와 원문 미저장
- 존재·미존재 로그인ID의 공통 오류와 더미 해시 수행
- 로그인·최초 등록 완료 세션ID 교체
- 쿠키 속성과 URL 세션ID 차단
- CSRF 없는 모든 상태 변경 요청 거부와 로그인 후 토큰 재발급
- 일반 세션 15분 유휴·8시간 절대 만료
- 제한 세션 10분 유휴·30분 절대 만료
- 계정 상태 또는 보안버전 불일치 시 다음 요청 차단
- 역할 메뉴권한 변경 시 해당 역할 사용자들의 보안버전 증가
- 중요 작업의 10분 재인증과 재인증 속도 제한
- 직접 URL·API·다른 사용자 자원 접근 거부
- 보안 응답·로그의 비밀정보 미포함
- 운영 보안 헤더와 캐시 금지
