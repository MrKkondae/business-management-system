# 데이터베이스 마이그레이션 규칙

## 1. 도구와 위치

- PostgreSQL 스키마 변경 도구는 Flyway로 확정한다.
- 파일은 백엔드의 `src/main/resources/db/migration`에서 관리한다.
- 애플리케이션은 시작 시 마이그레이션을 적용하고 JPA `ddl-auto=validate`로 매핑을 검증한다.
- 운영에서는 승인된 배포 절차가 사용하는 전용 권한으로만 DDL을 실행한다.

## 2. 파일 규칙

버전 파일명은 `VyyyyMMddHHmm__lower_snake_case_description.sql`을 사용한다. 같은 시각 충돌 시 분 단위를 조정한다. 예: `V202607211430__create_project_tables.sql`.

- 공유 환경에 한 번이라도 적용된 버전 파일은 수정·삭제·이름변경·버전 재사용하지 않는다.
- 잘못된 변경은 더 높은 버전의 보정 마이그레이션으로 수정한다.
- 반복 가능 `R__` 파일은 뷰·함수처럼 완전 재생성이 안전한 객체에만 사용하고 테이블·기준 데이터에는 사용하지 않는다.
- 한 파일은 하나의 목적을 가지며 가능한 한 트랜잭션 안에서 실행한다.
- 파괴적 변경은 컬럼 추가 → 이중 읽기/쓰기 또는 데이터 이행 → 검증 → 후속 배포에서 제거 순으로 분리한다.

## 3. 기준 데이터와 카탈로그

- 공통코드, 초기 역할·메뉴 등 실행에 필수인 기준 데이터는 멱등한 버전 마이그레이션으로 넣는다.
- 운영 사용자, 비밀번호, 개인 식별정보와 환경별 비밀값은 seed에 포함하지 않는다.
- 물리 모델 CSV를 먼저 변경하고 대응 마이그레이션과 JPA 매핑을 같은 변경 단위로 제출한다.
- CI에서 Flyway 검증, 빈 DB 전체 적용, 실제 스키마와 데이터 카탈로그의 테이블·컬럼·제약조건 일치를 검사한다.
- 대량 데이터 이행은 예상 잠금, 실행시간, 재시작·롤백 계획과 검증 SQL을 변경 설명에 포함한다.
