# 기능 설계 패키지

## 1. 목적

이 디렉터리는 하나의 사용자 가치 또는 유스케이스를 구현할 때 LLM과 개발자가 함께 읽어야 할 설계 원본과 변경 경계를 패키지 단위로 관리한다.

첫 번째 설계 묶음은 인증, 사용자, 역할·권한의 세 패키지로 구성한다. 세 패키지는 같은 사용자·역할·메뉴 데이터 모델을 사용하지만 구현 책임과 변경 경계를 분리한다.

## 2. 첫 번째 패키지 묶음

| 순서 | 패키지 ID | 디렉터리 | 백로그 | 기능 수 | 화면 | 상태 |
| --- | --- | --- | --- | ---: | --- | --- |
| 1 | `FPK-COM-AUTH-001` | `authentication/` | `PB-001` | 3 | `COM-001` | `DESIGN_IN_PROGRESS` |
| 2 | `FPK-SYS-USER-001` | `user-management/` | `PB-002` | 9 | `SYS-001` | `DESIGN_IN_PROGRESS` |
| 3 | `FPK-SYS-AUTHZ-001` | `role-authorization/` | `PB-018` | 9 | `SYS-002` | `DESIGN_IN_PROGRESS` |

각 디렉터리의 `manifest.yaml`은 기능·요구사항·백로그 추적, 설계 원본, 데이터 범위와 구현 시 변경 가능한 경로를 정의한다.

## 3. 경계 적용 규칙

- 경로는 저장소 루트를 기준으로 해석하고 `**`는 하위 경로 전체를 뜻한다.
- `allowed_paths`는 해당 패키지를 구현할 때 변경할 수 있는 최대 범위다. 실제 작업카드는 이보다 더 작은 파일 목록으로 제한한다.
- `forbidden_paths`는 패키지 구현 중 변경하지 않는 영역이다. 공통 계약 변경이나 다른 업무 모듈 변경이 필요하면 별도 작업과 ADR 검토를 선행한다.
- `database.allowed_objects`는 해당 패키지가 생성·조회·변경할 수 있는 테이블 범위다. 다른 테이블이 필요하면 데이터 아키텍처 원본을 먼저 변경한다.
- 공유 파일인 OpenAPI와 설계 카탈로그는 세 패키지가 함께 갱신할 수 있으므로 병렬 변경 시 충돌을 조정한다.
- Manifest 작성만으로 구현 준비 완료가 되지는 않는다. 기능·화면·API·테스트 상세설계와 카탈로그 연결이 모두 끝난 뒤 `READY`로 변경한다.

## 4. 확정 URL

| 화면 | URL | 소유 패키지 |
| --- | --- | --- |
| 로그인 | `/login` | `FPK-COM-AUTH-001` |
| 사용자관리 | `/system/users` | `FPK-SYS-USER-001` |
| 역할관리 | `/system/roles` | `FPK-SYS-AUTHZ-001` |

URL을 변경할 때는 화면 상세설계, 메뉴 기준 데이터와 관련 E2E 테스트를 함께 변경한다.

## 5. 후속 산출물

각 패키지에는 후속 작업에서 다음 산출물을 추가한다.

```text
{package}/
├── manifest.yaml
├── feature-spec.md
├── screens/
├── apis/
└── tests/
```

OpenAPI의 기계 판독 가능한 계약은 패키지 안에 복제하지 않고 `../../04.openapi/openapi.yaml`에 통합한다.
