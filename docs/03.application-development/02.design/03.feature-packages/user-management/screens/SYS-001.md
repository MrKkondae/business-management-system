## 2. 화면 목적

시스템관리자가 내부 직원의 최소정보, 사용자 계정과 초기 역할을 한 번에 등록하고
현재 사용자 목록을 검색·확인한다. 등록 성공 시 24시간 유효한 최초 등록 전용 임시
비밀번호를 한 번만 전달한다.

이번 단위에서는 목록 행을 선택해 상세 화면으로 이동하거나 기존 사용자를 변경하지
않는다. 화면에 표시되는 이메일과 휴대전화는 항상 서버에서 마스킹한 값만 사용한다.

## 3. 화면 책임과 경계

### 포함

- 로그인ID·사용자명·사원번호 통합 검색
- 계정상태·조직·역할 필터
- 허용 필드 정렬과 20·50·100건 페이징
- 미삭제 조직·역할 등록 옵션 조회
- 내부 직원 최소정보와 사용자 계정 동시 등록
- 복수 초기 역할 선택
- 임시 비밀번호·만료일시 1회 표시와 사용자 동작에 의한 복사
- 목록 개인정보 마스킹 표시
- 인증·권한·CSRF·검증·충돌·서버 오류 처리

### 포함하지 않음

- 사용자 상세 링크와 행 선택 동작
- 수정·삭제·사용중지·활성화 버튼
- 비밀번호 초기화와 재발급
- 원문 이메일·휴대전화 보기
- 엑셀 업로드·다운로드
- 일괄 선택과 일괄 처리
- 조직·역할 등록 또는 편집

### 3.1 사용자 등록 완성 단위

사용자 등록은 `TB_SYS_USER`만 생성하는 기능이 아니다. 다음 엔터티를 하나의
완성 단위로 생성한다.

```text
TB_COM_TASK_TGT
  TASK_TGT_ID
       │ 1:1
       ▼
TB_RES_MST
  RES_ID, TASK_TGT_ID, RES_SE_CD='EMPLOYEE'
       │ 공유 PK 1:1
       ▼
TB_RES_EMPLOYEE
  RES_ID, EMP_NO, ORG_ID, EMPLOY_STATUS_CD='EMPLOYED'
       │ 애플리케이션 참조
       ▼
TB_SYS_USER
  USER_ID, EMP_ID=RES_ID, LOGIN_ID, 계정·비밀번호 상태
       │ 1:N
       ▼
TB_SYS_USER_ROLE_REL
  USER_ID + ROLE_ID

같은 트랜잭션: TB_SYS_LOG(USER_CREATED)
```

- `TASK_TGT_ID`, `RES_ID`와 `USER_ID`는 서로 다른 식별자다.
- API의 `employeeId`와 `TB_SYS_USER.EMP_ID`는 직원 공유 PK인 `RES_ID`를 의미한다.
- `TB_RES_EMPLOYEE`는 `TB_RES_MST`와 `RES_ID`를 공유하는 직원 하위 엔터티다.
- 조직은 사용자 엔터티가 아니라 직원 엔터티의 `ORG_ID`에 저장한다.
- 역할은 사용자 엔터티 컬럼이 아니라 사용자–역할 관계로 저장한다.
- 업무대상은 v1에서 파일·메모를 구현하지 않아도 인력 공통 연결 기준을 위해
  함께 생성한다.

### 3.2 화면 입력과 엔터티 매핑

| 화면 입력·자동값 | 저장 대상 | 처리 기준 |
| --- | --- | --- |
| 사원번호 | `TB_RES_EMPLOYEE.EMP_NO` | 삭제 데이터 포함 전체 고유 |
| 성명 | `TB_RES_MST.FULL_NM`, `TB_SYS_USER.USER_NM` | 등록 시 같은 정리값 저장 |
| 조직 | `TB_RES_EMPLOYEE.ORG_ID` | 미삭제 조직 필수, 제출 시 재검증 |
| 로그인ID | `TB_SYS_USER.LOGIN_ID` | 소문자 정규화, 삭제 데이터 포함 전체 고유 |
| 초기 역할 | `TB_SYS_USER_ROLE_REL` | 서로 다른 역할 한 건 이상 |
| 이메일 | `TB_RES_MST.EMAIL_ADDR`, `TB_SYS_USER.EMAIL_ADDR` | 등록 시 같은 정리값 저장 |
| 휴대전화 | `TB_RES_MST.MOBILE_NO`, `TB_SYS_USER.MOBILE_NO` | 등록 시 같은 정리값 저장 |
| 업무대상유형 | `TB_COM_TASK_TGT.TASK_TGT_TYPE_CD` | 화면 입력 없이 `EMPLOYEE` |
| 인력구분 | `TB_RES_MST.RES_SE_CD` | 화면 입력 없이 `EMPLOYEE` |
| 입사일자 | `TB_RES_EMPLOYEE.HIRE_DT` | 이번 단위는 `NULL` |
| 직위 | `TB_RES_EMPLOYEE.JOB_POS_CD` | 이번 단위는 `NULL` |
| 재직상태 | `TB_RES_EMPLOYEE.EMPLOY_STATUS_CD` | 화면 입력 없이 `EMPLOYED` |
| 계정상태 | `TB_SYS_USER.ACNT_STATUS_CD` | 화면 입력 없이 `ACTIVE` |
| 초기 비밀번호 | `TB_SYS_USER.PWD_HASH_VAL` | 서버 난수 원문을 해시하여 저장 |
| 최초 등록 상태 | `PWD_INIT_REQ_YN`, `PWD_CHG_DTM`, `TEMP_PWD_EXPIRE_DTM` | `Y`, 발급시각, 발급시각+24시간 |
| 보안버전 | `TB_SYS_USER.SEC_VER` | `1` |

화면은 입사일자·직위·재직상태처럼 직원 상세관리에서 다룰 값을 임의 기본값으로
꾸며내지 않는다. 이번 수직 단위에 필요한 최소 필수값만 받고, 업무규칙으로 확정된
코드만 서버가 지정한다.

### 3.3 모듈 경계와 트랜잭션

```text
UserRegistrationService (@Transactional, system)
├── OrganizationQuery
├── RoleQuery
├── EmployeeRegistrationUseCase (employee 공개 인터페이스)
│   ├── TaskTargetRegistrationUseCase (common 공개 인터페이스)
│   ├── TB_RES_MST 생성
│   └── TB_RES_EMPLOYEE 생성
├── TB_SYS_USER 생성
├── TB_SYS_USER_ROLE_REL 생성
└── TB_SYS_LOG 생성
```

- 시스템관리 모듈은 `TB_RES_MST`, `TB_RES_EMPLOYEE`와 `TB_COM_TASK_TGT`를 직접
  변경하지 않는다.
- 인력 모듈 공개 인터페이스는 호출자의 로컬 트랜잭션에 참여하고 `RES_ID`를
  반환한다.
- 사용자·역할 또는 감사로그 생성이 실패하면 업무대상·인력·직원을 포함해 전부
  롤백한다.
- 사원번호와 로그인ID 중복은 사전 조회로 안내하되 최종 정합성은 DB 고유 인덱스로
  보장한다.
- 한 명의 미삭제 내부 직원에는 미삭제 사용자 계정을 최대 한 개만 연결한다.
- 조직과 직원, 직원과 사용자의 일부 참조는 애플리케이션 경계로 관리하므로 등록
  서비스와 통합 테스트에서 존재·활성·유형을 반드시 검증한다.

### 3.4 중복 개인정보와 기준 원본

성명·이메일·휴대전화는 현재 `TB_RES_MST`와 `TB_SYS_USER` 양쪽에 존재한다. 등록
시점에 같은 값을 저장하는 것만으로는 이후 정합성을 보장할 수 없다.

현재 최초 등록 구현은 선택 이메일·휴대전화를 `TB_SYS_USER`에만 반영하므로,
사용자가 최초 등록에서 값을 추가하거나 변경하면 `TB_RES_MST`와 값이 달라질 수
있다. 따라서 구현 전에 다음 중 하나를 명시적으로 선택해야 한다.

1. `TB_RES_MST`를 사람 개인정보의 기준 원본으로 정하고 사용자 생성·최초 등록·직원
   수정에서 두 엔터티를 같은 트랜잭션으로 동기화한다.
2. 두 컬럼의 의미를 인력 연락처와 계정 연락처로 분리하고 서로 달라도 정상으로
   정의한다.
3. 중복 컬럼을 제거하는 후속 데이터 모델 변경을 설계한다.

초안의 권장안은 1번이다. 시스템관리 모듈이 인력 테이블을 직접 수정하지 않도록
최초 등록도 인력 모듈의 개인정보 갱신 인터페이스를 호출해야 한다. 이 결정은
`D-012`로 관리한다.

- 정렬 가능한 열만 버튼 형태의 열 머리글과 현재 정렬 방향을 제공한다.

- 데스크톱에서는 우측 고정 패널, 모바일에서는 전체 화면 패널을 권장한다.
- 패널은 하나의 명확한 제목을 가지며 열릴 때 첫 입력인 사원번호로 포커스를 옮긴다.
- 패널 밖의 목록은 `inert` 또는 동등한 포커스 차단을 적용한다.
- `Escape`, 닫기와 취소는 입력 변경이 없을 때 패널을 닫는다.
- 입력 변경이 있으면 “작성 중인 내용이 사라집니다” 확인 후 닫는다.
- 등록 처리 중에는 닫기·취소·입력·제출을 비활성화한다.

등록 패널 방식은 `D-001` 확정 대상이다.

### 4.3 임시 비밀번호 완료 대화상자

```text
┌──────────────────────────────────────────────┐
│ 사용자 등록 완료                             │
│                                              │
│ 로그인ID       staff01              [복사]  │
│ 임시 비밀번호  X7...한 번만 표시...  [복사]  │
│ 만료일시       2026-07-25 14:30              │
│                                              │
│ 이 창을 닫으면 임시 비밀번호를 다시          │
│ 조회할 수 없습니다. 안전한 방법으로 전달하세요.│
│                                              │
│                         [확인하고 닫기]       │
└──────────────────────────────────────────────┘
```

- 등록 응답을 받은 뒤 등록 패널을 완료 대화상자로 교체한다.
- 로그인ID, 임시 비밀번호와 만료일시를 표시한다.
- 임시 비밀번호는 기본적으로 가리지 않는다. 이미 1회 전달을 위한 전용 단계이며
  사용자가 정확히 복사·전달할 수 있어야 한다.
- 복사는 사용자가 버튼을 누른 경우에만 수행하며 자동 복사하지 않는다.
- 복사 성공은 “복사되었습니다” 상태 텍스트로 알리고 원문을 로그에 남기지 않는다.
- 대화상자는 바깥 영역 클릭이나 `Escape`로 닫지 않는다.
- `확인하고 닫기`로만 닫고 이후 임시 비밀번호 상태를 즉시 폐기한다.
- 닫은 뒤 현재 조회조건을 유지하고 목록 첫 페이지를 재조회한다.
- 새 사용자가 현재 필터에 포함되지 않더라도 등록 성공 안내는 유지한다.

- 테이블을 가로 스크롤시키지 않고 동일 데이터를 카드 목록으로 표시하는 방안을
  권장한다.
- 필터는 접을 수 있는 영역으로 제공하되 적용된 필터 수와 초기화 동작을 노출한다.
- 등록 패널은 화면 전체를 사용하며 모바일 키보드가 제출 버튼을 영구히 가리지 않게
  세로 스크롤을 허용한다.
- 본문 좌우 여백은 최소 16px, 버튼과 입력 높이는 최소 44px로 한다.
- 320px 너비에서 정보 손실과 문서 전체의 가로 스크롤이 없어야 한다.

모바일 목록 표현은 `D-003` 확정 대상이다.

## 6. 화면 진입과 권한

1. 공통 `AuthProvider`가 `GET /auth/me`로 일반 세션을 확인한다.
2. 미인증이면 `/login`, 제한 세션이면 `/account/initial-registration`으로 이동한다.
3. 일반 세션에 `SYS-001` 메뉴 권한이 없으면 `403` 전용 안내를 표시한다.
4. 권한이 있으면 URL query를 조회조건 상태로 해석한다.
5. `API-SYS-USER-001` 목록과 `API-SYS-USER-003` 옵션을 병렬 조회한다.
6. 목록 성공·옵션 실패 시 목록은 표시하고 조직·역할 필터 및 사용자 등록은
   비활성화한 뒤 옵션 재시도를 제공한다.
7. 목록 실패 시 기존 목록이 없으면 목록 영역에 오류와 재시도를 표시한다.

프런트엔드 메뉴 표시와 직접 URL 차단은 사용성 보조수단이며 서버의 `SYS-001`
권한 검사를 대체하지 않는다.

## 7. 조회조건과 URL query

| 화면 항목 | Query parameter | 초기값 | 입력·전송 규칙 |
| --- | --- | --- | --- |
| 검색어 | `keyword` | 없음 | 앞뒤 공백 제거, 빈 값은 query에서 제거 |
| 계정상태 | `accountStatusCode` | 전체 | `ACTIVE`, `INACTIVE`; 전체는 query에서 제거 |
| 조직 | `organizationId` | 전체 | 옵션의 조직ID; 전체는 query에서 제거 |
| 역할 | `roleId` | 전체 | 옵션의 역할ID; 전체는 query에서 제거 |
| 페이지 | `page` | `0` | API·URL 모두 0 기반, 화면 번호는 1 기반 |
| 페이지 크기 | `size` | `20` | `20`, `50`, `100`만 허용 |
| 정렬 | `sort` | `registeredAt,desc` | 허용 필드와 `asc`·`desc` |

허용 정렬 필드는 다음과 같다.

| 화면 열 | 정렬값 |
| --- | --- |
| 등록일시 | `registeredAt` |
| 사용자명 | `userName` |
| 로그인ID | `loginId` |
| 사원번호 | `employeeNumber` |
| 마지막로그인 | `lastLoginAt` |

- 입력 중인 검색어·필터와 실제 적용 조건을 구분한다.
- `조회` 또는 검색어 입력 중 `Enter`를 실행하면 적용 조건을 URL에 반영하고
  `page=0`으로 조회한다.
- `초기화`는 검색어·필터·정렬·페이지 크기를 화면 초기값으로 되돌리고 조회한다.
- 정렬 변경, 페이지 이동과 페이지 크기 변경도 URL에 반영한다.
- 페이지 크기를 변경하면 `page=0`으로 이동한다.
- URL의 알 수 없는 값은 초기값으로 교정하고 서버에 그대로 전달하지 않는다.
- 뒤로가기·앞으로가기·새로고침 시 URL 기준 상태와 목록을 복원한다.

- 상태는 텍스트를 포함하며 색상만으로 구분하지 않는다.
- 서버가 반환한 마스킹 연락처를 재가공해 원문처럼 보이게 하지 않는다.
- 사용자ID와 직원ID는 화면에 표시하지 않고 향후 상세 이동 키로만 보존한다.
- 이번 단위에서는 행 hover를 클릭 가능한 형태로 표현하지 않는다.

## 9. 목록 로딩·빈 상태·페이징

| 상태 | 표시와 동작 |

| 최초 조회 | 테이블·카드 형태의 스켈레톤과 조회 중 상태 표시 |
| 조건 변경 조회 | 기존 목록 유지, 목록 영역 진행 상태와 중복 조회 차단 |
| 결과 0건 | “조회 결과가 없습니다”와 조건 초기화 제공 |

| 마지막 페이지 | 다음 버튼 비활성화 |
| 전체 0건 | 페이지 번호 대신 `0건` 표시 |

- 모바일은 `현재 페이지 / 전체 페이지`와 이전·다음만 표시한다.
- 현재 페이지가 삭제나 조건 변화로 전체 페이지 범위를 벗어나면 마지막 유효
  페이지로 한 번 교정하여 재조회한다.

## 10. 등록 입력

| 필드 | HTML/표현 | 필수 | 클라이언트 검증 | 제출 처리 |
| --- | --- | :---: | --- | --- |
| 사원번호 | `text` | Y | 앞뒤 공백 제거 후 1~30자 | 정리된 값 전송 |
| 성명 | `text`, `name` | Y | 앞뒤 공백 제거 후 1~100자 | 정리된 값 전송 |
| 조직 | 단일 선택 | Y | 옵션 한 건 선택 | `organizationId` 전송 |
| 로그인ID | `text`, `username` | Y | 정리 후 4~26자, 영문자·숫자·점·밑줄·하이픈, 첫 글자 영문자 또는 숫자 | 소문자 변환 후 전송 |
| 초기 역할 | 체크박스 목록 | Y | 서로 다른 역할 한 건 이상 | `initialRoleIds` 배열 전송 |
| 이메일 | `email`, `email` | N | 정리 후 최대 100자, 값이 있으면 이메일 형식 | 빈 값은 필드 생략 |
| 휴대전화 | `tel`, `tel` | N | 정리 후 최대 20자, 숫자와 하이픈만 허용 | 빈 값은 필드 생략 |

- 필수 표시는 레이블 텍스트와 접근성 속성으로 제공한다.
- 로그인ID 입력 중 소문자 정규화 예정값을 바로 보여주거나 입력값 자체를 소문자로
  변환한다. 저장값이 달라질 수 있음을 숨기지 않는다.
- 역할은 서버 옵션 순서를 유지하고 체크박스 그룹 이름을 제공한다.
- 조직이나 역할 옵션이 비어 있으면 저장을 차단하고 “기준정보가 필요합니다”를
  안내한다. 이번 화면에서 기준정보 등록 링크는 제공하지 않는다.
- 등록 패널을 열 때 옵션을 다시 조회해 삭제된 기준정보 선택 가능성을 낮춘다.
- 서버는 제출 시점의 조직·역할 상태를 다시 검증한다.

선택 필드의 빈 값 JSON 표현은 `D-007` 확정 대상이다.

## 11. 등록 처리

1. 사용자가 `사용자 등록`을 선택한다.
2. 등록 옵션을 재조회하고 성공하면 패널과 첫 입력 포커스를 연다.
3. 제출 시 필수·형식·역할 중복을 검증한다.
4. 오류가 있으면 필드 인접 메시지를 표시하고 첫 오류 필드로 포커스를 이동한다.
5. 유효하면 모든 입력과 패널 닫기·등록 버튼을 잠근다.
6. `API-SYS-USER-002`를 한 번 호출한다.
7. 성공하면 비밀번호를 폼 상태에서 분리해 완료 대화상자에만 보관한다.
8. 사용자가 `확인하고 닫기`를 선택하면 임시 비밀번호를 즉시 폐기한다.
9. 현재 조회조건을 유지하고 목록 `page=0`을 재조회한다.

중복 클릭, `Enter` 연속 입력과 느린 네트워크가 등록 요청을 두 건 만들지 않아야
한다. 등록 성공 후 같은 응답을 다시 표시하거나 임시 비밀번호를 재조회하지 않는다.

## 12. 등록 오류 처리

| 오류/상태 | 화면 처리 |
| --- | --- |
| 클라이언트 검증 실패 | 필드 인접 메시지와 첫 오류 포커스 |
| `COMMON_VALIDATION_FAILED` | `fieldErrors`를 해당 필드와 연결 |
| `SYS_LOGIN_ID_DUPLICATE` | 로그인ID 필드에 “이미 사용 중인 로그인ID입니다.” |
| `RES_EMPLOYEE_NUMBER_DUPLICATE` | 사원번호 필드에 “이미 사용 중인 사원번호입니다.” |
| `COMMON_RESOURCE_NOT_FOUND` | 조직 옵션을 재조회하고 조직 필드에 재선택 안내 |
| `COMMON_INVALID_STATE` | 조직·역할 옵션을 재조회하고 변경된 항목 재선택 안내 |
| `AUTH_CSRF_INVALID` | 새 CSRF 토큰을 받아 잠긴 동일 제출을 한 번만 재시도 |
| `AUTH_AUTHENTICATION_REQUIRED` | 인증 메모리 제거 후 `/login` 이동 |
| `AUTH_PASSWORD_CHANGE_REQUIRED` | `/account/initial-registration` 이동 |
| `AUTH_FORBIDDEN` | 입력과 응답을 폐기하고 403 안내 표시 |
| 네트워크 오류 | 입력값 유지, 안전한 연결 오류와 재시도 제공 |
| 서버 오류 | 입력값 유지, 안전한 메시지와 `traceId` 표시 |

- 실패 시 임시 비밀번호가 생성되었는지 추측하거나 표시하지 않는다.
- 중복 오류 후 다른 입력값은 유지한다.
- 조직·역할 재조회로 선택값이 사라지면 해당 선택만 해제하고 이유를 안내한다.
- 등록 성공 응답을 받은 뒤 완료 대화상자 렌더링에 실패하더라도 응답 원문을
  콘솔·브라우저 저장소·오류수집 서비스에 기록하지 않는다.

## 13. 임시 비밀번호 보안

- 임시 비밀번호는 등록 요청을 처리하는 컴포넌트의 메모리에만 일시 보관한다.
- URL, `localStorage`, `sessionStorage`, 쿠키, 전역 상태와 분석 이벤트에 저장하지
  않는다.
- React 개발도구나 오류 객체에 불필요하게 남지 않도록 완료 상태를 최소화한다.
- 네트워크 응답, 요청 헤더와 임시 비밀번호를 콘솔에 출력하지 않는다.
- 복사 버튼은 Clipboard API를 사용하되 권한 거부 시 수동 선택 방법을 안내한다.
- 창 닫기·새로고침·경로 이동 후 복원하지 않는다.
- 완료 대화상자를 닫은 뒤 문자열 참조를 제거한다.
- 임시 비밀번호가 포함된 화면에 `Cache-Control: no-store` 계약을 유지한다.
- 복사 성공 여부는 기록할 수 있지만 복사된 문자열은 어떤 로그에도 포함하지 않는다.

## 14. API 연결

| 순서 | API ID | Method·Path | 목적 |
| ---: | --- | --- | --- |
| 1 | `API-SYS-USER-001` | `GET /users` | 사용자 목록·검색·필터·정렬·페이징 |
| 2 | `API-SYS-USER-003` | `GET /users/registration-options` | 조직·역할 필터와 등록 옵션 |
| 3 | `API-SYS-USER-002` | `POST /users` | 직원·사용자·초기 역할 등록과 임시 비밀번호 1회 발급 |

브라우저는 `/backend-api` 동일 출처 프록시를 사용한다. 화면에서 직접 `fetch`하지
않고 `shared/api`의 사용자관리 클라이언트를 통해 호출한다.

요청·응답 필드, HTTP 상태와 오류코드는 OpenAPI와 `apis.csv`를 원본으로 한다.

## 15. 접근성과 키보드

- 페이지 제목은 “사용자관리 | BMS”로 제공하고 하나의 명확한 `h1`을 둔다.
- 검색 영역은 이름 있는 `form`, 목록은 제목과 총건수를 연결한다.
- 열 정렬 버튼은 정렬 가능 여부와 `aria-sort`를 제공한다.
- 로딩은 `role="status"`, 오류와 복사 완료는 적절한 live region으로 알린다.
- 등록 패널은 대화상자 이름, 포커스 진입·순환·복귀를 보장한다.
- 패널을 닫으면 `사용자 등록` 버튼으로 포커스를 돌린다.
- 완료 대화상자를 닫으면 목록 제목 또는 등록 버튼으로 포커스를 돌린다.
- 필드 오류는 `aria-describedby`, 필수 여부는 `aria-required`와 보이는 표시로
  연결한다.
- 체크박스 역할 목록은 `fieldset`과 `legend`를 사용한다.
- 마우스 없이 조회조건 입력, 정렬, 페이지 이동, 등록, 복사와 닫기가 가능해야 한다.
- 상태와 정렬 방향을 색상·아이콘에만 의존하지 않는다.

## 16. 개인정보와 표시 형식

- 이메일과 휴대전화는 API가 반환한 마스킹값만 표시한다.
- 목록 화면에서 원문 전체보기, 툴팁 원문과 HTML 속성 원문을 제공하지 않는다.
- 날짜·시간은 UTC 응답을 Asia/Seoul `YYYY-MM-DD HH:mm`으로 표시한다.
- 등록 폼의 원문 이메일·휴대전화는 제출에만 사용하고 성공 후 즉시 비운다.
- 사용자ID·직원ID·연락처 원문을 분석 이벤트와 클라이언트 진단로그에 기록하지
  않는다.
- 임시 비밀번호, 비밀번호 해시, 세션ID, 쿠키와 CSRF 토큰은 어떤 화면 로그에도
  기록하지 않는다.

## 17. 화면 전이

```text
/login
  └─ 일반 로그인 성공 + SYS-001 구현 메뉴 권한
       └─ /system/users

/system/users
  ├─ 사용자 등록 → 등록 패널
  │                    ├─ 실패 → 입력 유지
  │                    └─ 성공 → 임시 비밀번호 완료 대화상자
  │                                  └─ 확인하고 닫기 → 목록 page=0 재조회
  ├─ 세션 만료 → /login
  ├─ 제한 세션 → /account/initial-registration
  └─ 권한 없음 → 403 안내
```

첫 구현 메뉴 자동 이동 정책에 따라 `/system/users`가 프런트엔드 구현 메뉴 목록에
등록되면 해당 권한 사용자의 로그인 후 첫 진입점이 될 수 있다.

## 18. 인수조건

- URL query로 조회조건·페이지·크기·정렬이 새로고침과 뒤로가기 후 복원된다.
- 조회 실행 시 첫 페이지로 이동하고 페이지 크기 변경도 첫 페이지로 이동한다.
- 조건 변경 조회 중 기존 목록을 임의로 비우지 않는다.
- 이메일과 휴대전화 원문이 목록·HTML·로그에 노출되지 않는다.
- 정렬 가능한 열만 키보드로 정렬할 수 있고 현재 방향을 알 수 있다.
- 결과 0건은 정상 빈 상태와 조건 초기화를 제공한다.
- 등록 필수·형식 오류는 API를 호출하지 않고 첫 오류 필드로 포커스한다.
- 등록 중 클릭과 `Enter`가 중복 요청을 만들지 않는다.
- 로그인ID는 소문자로 정규화된 값을 명확히 표시하고 전송한다.
- 초기 역할을 한 건 이상 선택해야 하며 중복 역할ID를 전송하지 않는다.
- 중복 로그인ID·사원번호 오류가 해당 필드에 표시되고 나머지 입력은 유지된다.
- 조직·역할이 삭제되거나 변경되면 옵션을 재조회하고 다시 선택하도록 안내한다.
- 등록 성공 시 임시 비밀번호와 만료일시를 한 번만 표시한다.
- 완료 대화상자를 닫은 뒤 임시 비밀번호를 복원하거나 재조회할 수 없다.
- 권한 없는 사용자의 직접 URL 접근과 API 호출이 모두 거부된다.
- 모바일 320px에서 문서 전체의 가로 스크롤과 핵심 정보 손실이 없다.
- 키보드만으로 검색·정렬·페이징·등록·복사·닫기를 완료할 수 있다.

## 19. 구현 파일 구조 초안

```text
frontend/
├── app/
│   └── (main)/system/users/
│       └── page.tsx
├── features/
│   └── system/users/
│       ├── UserManagementClient.tsx
│       ├── UserSearchForm.tsx
│       ├── UserList.tsx
│       ├── UserPagination.tsx
│       ├── UserRegistrationPanel.tsx
│       ├── TemporaryPasswordDialog.tsx
│       ├── user-search-params.ts
│       └── user-registration-validation.ts
└── shared/
    ├── api/
    │   └── users-api.ts
    └── types/
        └── users.ts
```

- `page.tsx`는 메타데이터와 화면 조합만 담당한다.
- URL query 해석·직렬화는 `user-search-params.ts` 한 곳에서 처리한다.
- HTTP 호출은 `shared/api/users-api.ts`에서만 수행한다.
- 임시 비밀번호 타입은 등록 성공 처리 경계를 넘어 전역 인증 상태에 포함하지 않는다.
- 작은 표시 컴포넌트는 사용처에 두고 실제 재사용 전까지 공통화하지 않는다.

있지만, 인수조건과 테스트 케이스가 달라지므로 함께 확정하는 것이 좋다.
