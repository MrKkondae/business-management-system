# 사용자 관리 1차 단위 기능 명세 초안

## 1. 문서 정보

| 항목 | 내용 |
| --- | --- |
| 기능 패키지 | `FPK-SYS-USER-001` |
| 작업 단위 | 사용자 목록·등록 및 최초 등록 |
| 백로그 | `PB-002` 사용자 관리 |
| 요구사항 | `REQ-SYS-001`, `REQ-RES-001`, `NFR-SEC-001`, `NFR-SEC-003` |
| 화면 | `SYS-001`, `SYS-009` |
| 상태 | 초안 (`DESIGN_IN_PROGRESS`) |

이 문서는 사용자 관리 전체 기능 중 사용자 목록·등록 단위와 별도로 구현 준비가 완료된 최초 등록·초기 비밀번호 변경 단위를 함께 정의한다. `SYS-001`은 계속 설계 중이며 `SYS-009`와 `API-SYS-USER-004`의 설계 상태는 `READY`이다.

## 2. 범위

### 포함

- 사용자 목록 검색·필터·페이징
- 등록 가능한 미삭제 조직과 역할 옵션 조회
- 사원번호, 성명, 조직, 로그인ID와 초기 역할 입력
- 선택 이메일·휴대전화 입력
- 업무대상, 인력 공통정보, 직원, 사용자와 사용자역할 관계 생성
- 24시간 유효한 최초 등록 전용 임시 비밀번호 발급
- 사용자 생성 시스템로그와 개인정보 마스킹

### 제외

- 사용자 상세·수정·사용중지와 계정 활성·비활성
- 비밀번호 초기화·변경과 최초 등록 완성 화면 구현
- 최초 시스템관리자 부트스트랩
- 외주인력 계정, 공개 회원가입과 직원 일괄등록
- 역할·조직 기준정보의 등록·수정·삭제

## 4. 사용자와 권한

- `SYS-001` 메뉴 권한이 있는 시스템관리자만 API와 화면에 접근할 수 있다.
- 프론트엔드의 메뉴·버튼 숨김과 관계없이 모든 API에서 세션의 `SYS-001` 권한을 검사한다.
- 등록 처리자ID, 요청추적ID와 신뢰 프록시 기준 접속 IP를 시스템로그에 기록한다.

| `USER-BR-004` | 사원번호는 앞뒤 공백을 제거한 1~30자이며 삭제된 직원을 포함하여 중복될 수 없다. |
| `USER-BR-005` | 미삭제 조직 한 건과 미삭제 역할 한 건 이상을 지정해야 한다. |

| `USER-BR-007` | 신규 사용자는 `ACTIVE`, 로그인실패건수 0, 비밀번호초기화필요여부 `Y`, 삭제여부 `N`으로 생성한다. |
| `USER-BR-008` | 임시 비밀번호는 암호학적 난수로 16자 생성하고 비밀번호 정책의 문자종류 3종 이상을 만족한다. 원문은 생성 성공 응답에서 한 번만 제공하고 이후 재조회하지 않는다. |
| `USER-BR-009` | 임시 비밀번호는 발급 시각인 `PWD_CHG_DTM`부터 24시간 동안 최초 등록 제한 세션 로그인에만 사용할 수 있으며 `TEMP_PWD_EXPIRE_DTM`에 만료일시를 저장한다. 최초 등록 완료 전에는 유효기간 안에서 다시 로그인할 수 있고 완료 또는 재발급 즉시 이전 값을 폐기한다. |
| `USER-BR-010` | 직원·사용자·역할 관계·시스템로그는 하나의 트랜잭션에서 생성하며 일부 성공 데이터를 남기지 않는다. |
| `USER-BR-011` | 시스템 모듈은 인력 테이블을 직접 변경하지 않고 인력 모듈의 공개 등록 인터페이스를 호출한다. |
| `USER-BR-012` | 사용자명과 선택 연락처의 최초 값은 인력정보와 사용자 계정에 동일하게 기록하며 이 단위 안에서 불일치를 허용하지 않는다. |
| `USER-BR-013` | 사용자관리 1차 등록은 신규 내부 직원과 사용자 계정을 함께 생성한다. 이미 존재하는 사원번호는 중복으로 거부하며 기존 직원 연결은 후속 단위로 분리한다. |
| `USER-BR-014` | 성명·이메일·휴대전화의 기준 원본은 인력정보다. 사용자 계정의 동명 값은 인증·표시용 사본이며 인력 모듈 공개 인터페이스를 통해 같은 트랜잭션으로 동기화한다. |
| `USER-BR-015` | 1차 등록은 입사일자와 직위를 받지 않고 `NULL`로 저장한다. 인력구분은 `EMPLOYEE`, 재직상태는 `EMPLOYED`로 생성한다. |

## 6. 사용자 목록 조회

### 조회조건

| 조건 | 처리 |
| --- | --- |
| 검색어 | 로그인ID, 사용자명과 사원번호의 부분 일치 |
| 계정상태 | `ACTIVE`, `INACTIVE` |
| 조직 | 조직ID 정확 일치 |
| 역할 | 역할ID가 부여된 사용자 |

- 삭제되지 않은 사용자만 조회한다.
- 기본 정렬은 등록일시 내림차순이며 허용 정렬은 등록일시, 사용자명, 로그인ID, 사원번호와 마지막로그인일시다.
- 기본 20건, 최대 100건의 0 기반 페이지를 사용한다.
- 휴대전화는 `010-****-1234`, 이메일은 `ab***@domain.com` 형식으로 마스킹한다.
- 역할이 여러 개이면 역할명을 중복 없이 표시한다.

## 7. 사용자 등록 처리

1. `SYS-001` 권한과 CSRF 토큰을 검증한다.
2. 입력 형식과 필수값을 검증하고 로그인ID를 소문자로 정규화한다.
3. 미삭제 조직과 미삭제 초기 역할을 조회한다.
4. 로그인ID와 사원번호 중복을 사전 확인한다.
5. 임시 비밀번호 원문을 생성하고 승인된 `PasswordEncoder`로 해시한다.
6. 인력 모듈의 `EmployeeRegistrationUseCase`를 호출한다. 사원번호·성명·조직과 선택 연락처를 전달하고 입사일자·직위는 `NULL`, 인력구분·재직상태는 각각 `EMPLOYEE`, `EMPLOYED`로 생성한다.
7. 인력 모듈은 `TB_COM_TASK_TGT`, `TB_RES_MST`, `TB_RES_EMPLOYEE`를 순서대로 생성하고 직원ID를 반환한다.
8. 시스템 모듈은 반환된 직원ID로 `TB_SYS_USER`를 생성하면서 `PWD_INIT_REQ_YN='Y'`, `PWD_CHG_DTM=발급시각`, `TEMP_PWD_EXPIRE_DTM=발급시각+24시간`을 저장하고 선택한 `TB_SYS_USER_ROLE_REL`을 생성한다.
9. `USER_CREATED` 시스템로그를 같은 트랜잭션에서 생성한다.
10. 커밋이 완료된 후 임시 비밀번호와 만료일시를 응답에서 한 번만 반환한다.

중복 사전 확인은 사용자 안내를 위한 것이며 정합성 보장의 최종 기준은 DB 고유 인덱스다. 동시 요청에서 고유성 충돌이 발생하면 해당 필드의 중복 오류로 변환하고 전체 트랜잭션을 롤백한다.

## 8. 모듈 경계와 트랜잭션

```text
UserRegistrationService (@Transactional)
├── OrganizationQuery
├── RoleQuery
├── EmployeeRegistrationUseCase
│   ├── 업무대상 생성
│   ├── 인력 공통정보 생성
│   └── 직원정보 생성
├── SystemUserRepository
├── UserRoleRepository
└── SystemLogRepository
```

- v1.0은 단일 Spring Boot 애플리케이션과 동일 PostgreSQL DataSource를 사용하므로 하나의 로컬 트랜잭션으로 처리한다.
- 인력 모듈 공개 인터페이스의 구현은 호출자의 트랜잭션에 참여하며 별도 비동기 처리나 독립 커밋을 하지 않는다.
- 임시 비밀번호 원문은 트랜잭션이 실패하면 폐기하고 어떤 로그에도 기록하지 않는다.

| `TB_SYS_USER_ROLE_REL` | 조회·생성 | system | 목록 역할과 초기 역할 부여 |
| `TB_SYS_LOG` | 생성 | system | `USER_CREATED` 감사 사건 |

## 10. 오류 계약

외부 오류코드 정의는 `../../02.catalog/error-codes.csv`, 사용자 관리 API별 오류 연결은 `../../02.catalog/apis.csv`를 원본으로 한다. 조직·역할 없음, 정규화 로그인ID·사원번호 중복과 삭제된 기준정보 선택은 등록 트랜잭션을 시작하거나 커밋하지 않는다.

## 11. 개인정보와 로그

- 목록의 이메일과 휴대전화는 항상 마스킹하고 이 단위에서는 원문 전체보기 API를 제공하지 않는다.
- 임시 비밀번호, 비밀번호 해시와 연락처 변경 전후 값은 시스템로그와 진단로그에 기록하지 않는다.
- `USER_CREATED` 로그에는 처리자ID, 생성된 사용자ID, `USER` 대상유형, 성공 결과, 추적ID와 접속 IP만 기록한다.
- 등록 실패 시 데이터와 성공 시스템로그를 모두 롤백하며 진단로그에는 안전한 오류코드와 추적ID만 기록한다.

## 12. 완료 조건

- 목록·등록 옵션·등록 API와 SYS-001 화면 계약이 일치한다.
- 직원·사용자·초기 역할 생성의 트랜잭션과 롤백 시나리오가 정의된다.
- 로그인ID·사원번호의 사전 확인과 동시 중복 요청을 모두 검증한다.
- 임시 비밀번호가 한 번만 노출되고 로그·재조회에서 노출되지 않음을 검증한다.
- 임시 비밀번호 만료일시가 발급시각부터 24시간으로 저장되고 초기화필요여부와 DB 제약조건이 일치함을 검증한다.
- 개인정보 마스킹, 미인증 401, 무권한 403과 CSRF 실패를 검증한다.

## 13. 후속 단위

- 사용자 상세·수정·사용중지
- 관리자 비밀번호 초기화
- 계정 활성·비활성 및 로그인실패 비활성 해제
- 최초 시스템관리자 부트스트랩 실행 절차

최초 등록 완성 단위는 제한 세션의 계정 상태·초기화필요여부·임시 비밀번호 만료·로그인 당시 `PWD_CHG_DTM`을 다시 검증한다. 완료 시 새 비밀번호 정책과 현재 임시 비밀번호 불일치를 검증하고 새 비밀번호 해시, 선택 개인정보, `PWD_INIT_REQ_YN='N'`, `TEMP_PWD_EXPIRE_DTM=NULL`을 한 트랜잭션으로 저장한 뒤 세션ID 교체 후 새 역할·메뉴 스냅샷의 일반 세션으로 승격한다. 선택 개인정보가 있으면 인력 모듈의 개인정보 갱신 인터페이스를 호출하여 `TB_RES_MST`와 `TB_SYS_USER`를 같은 트랜잭션으로 동기화한다. 현재 `TB_SYS_USER`만 변경하는 구현은 이 기준에 맞게 보완해야 한다. 화면 계약은 `screens/SYS-009.md`, 공통 프런트엔드 계약은 인증 패키지의 `frontend-implementation-contract.md`를 따른다.
