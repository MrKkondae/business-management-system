# 인증·보안 테스트 구현 계획

## 1. 목적과 원본

인증 기능의 정책 문장만 확인하지 않고 비밀번호, 세션, CSRF, 권한, 재인증, 로그와 초기 부트스트랩이 실제 구현 경계에서 함께 동작하는지 검증한다.

- 사용자 관점 인수조건: `acceptance.feature`
- API ID·오류·권한 연결: `../../../02.catalog/apis.csv`
- 외부 오류 정의: `../../../02.catalog/error-codes.csv`
- 권한 정의: `../../../02.catalog/permissions.csv`
- 기계 판독 API 계약: `../../../04.openapi/openapi.yaml`
- 보안 구현 기준: `../../../01.common/09.security-implementation-contract.md`

## 2. 공통 테스트 제어

- 운영 코드의 현재시각 조회는 주입 가능한 `java.time.Clock` 경계로 모은다.
- 만료 테스트에서 `Thread.sleep`과 실제 분·시간 대기를 사용하지 않는다.
- 모든 만료시간은 `기준시각-1ms`, `기준시각`, `기준시각+1ms`를 검증하며 정확한 기준시각부터 만료로 판정한다.
- 로그인 속도 제한기는 테스트용 수동 시계와 격리된 저장소를 사용한다.
- 동시성 테스트는 실제 PostgreSQL Testcontainers에서 서로 다른 트랜잭션과 연결을 사용한다.
- 비밀번호·쿠키·CSRF·비밀키 fixture는 테스트 전용 값을 사용하고 실패 메시지에도 원문을 출력하지 않는다.
- 각 테스트는 독립된 사용자·세션·접속로그를 만들고 실행순서와 공유 상태에 의존하지 않는다.

## 3. 단위 테스트

| 대상 | 필수 검증 |
|---|---|
| 로그인ID 정규화 | 앞뒤 공백 제거, 영문 소문자 변환, 이미 정규화된 값의 멱등성 |
| 비밀번호 정책 | 12·64자 경계, 문자종류 3종, 금지어·로그인ID·성명 포함, 최근 5개 재사용 |
| Argon2id 인코더 | 알고리즘과 최소 파라미터, 사용자별 salt, 정상·오류 비교, 재해시 필요 판정 |
| 로그인 실패 상태전이 | 1~5회 ACTIVE, 6회 INACTIVE·LOGIN_FAILURE·보안버전 증가, 성공 시 0 |
| 세션 유효성 판정 | 일반 15분/8시간, 제한 10분/30분/임시 비밀번호 만료, 가장 빠른 만료 우선 |
| 활동 판정 | 활동 API만 `lastUserActivityAt` 갱신, 절대 만료 불변 |
| 제한 세션 허용목록 | 허용 5개 API와 나머지 `AUTH_PASSWORD_CHANGE_REQUIRED` |
| 보안버전 판정 | 동일 버전 허용, 불일치 세션 종료, 음수·0 저장 거부 |
| 권한 합집합 | 중복 메뉴 제거, 삭제 역할·메뉴 제외, 상위 메뉴 표시정보 유지 |
| 재인증 | 성공시각 기록, 10분 경계, 비밀번호 실패 시 시각·로그인실패건수 불변 |
| 속도 제한 | 로그인ID·IP 5회, IP 30회, 재인증 세션 5회와 60초 해제 경계 |
| 로그 정제 | 비밀번호·해시·세션ID·쿠키·CSRF·Authorization 헤더 제거 |

## 4. 저장소·마이그레이션 테스트

| 대상 | 필수 검증 |
|---|---|
| 빈 DB | 모든 Flyway 마이그레이션을 처음부터 적용하고 인증 MyBatis Mapper 통합 테스트 통과 |
| 기존 버전 DB | 기존 마이그레이션 체크섬을 바꾸지 않고 후속 버전으로 업그레이드 |
| 사용자 제약 | 로그인ID 소문자 저장, `SEC_VER >= 1`, 계정상태·비활성정보, 임시 비밀번호 일관성 CHECK |
| 유일성 | 대소문자 정규화 로그인ID와 사원번호 동시 등록에서 한 건만 성공 |
| 선행데이터 | 기본 역할·메뉴·공통코드 생성과 시스템관리자의 모든 유효 메뉴 관계 |
| 권한 변경 | 역할 사용자 전체의 `SEC_VER` 증가가 권한 변경과 같은 트랜잭션으로 커밋·롤백 |
| 접속로그 | 성공·실패 필드 제약, TIMEOUT 중복 갱신 방지, 요청추적ID 연결 |

## 5. API 통합 테스트

`apis.csv`의 모든 API는 다음 공통 조합을 자동 생성하거나 파라미터화하여 검증한다.

| 구분 | 기대 결과 |
|---|---|
| 정상 권한·정상 요청 | 카탈로그와 OpenAPI에 정의된 2xx |
| 필수값·형식 오류 | 등록된 400 오류와 `Problem` 구조 |
| 세션 없음·유휴/절대 만료 | `401 AUTH_AUTHENTICATION_REQUIRED` |
| 비활성·삭제·보안버전 불일치 | 업무 미실행, 세션 종료, 401 |
| 메뉴·데이터 권한 없음 | `403 AUTH_FORBIDDEN` |
| 제한 세션의 일반 API | `403 AUTH_PASSWORD_CHANGE_REQUIRED` |
| 상태 변경 CSRF 없음·오류 | `403 AUTH_CSRF_INVALID`, DB 변경 없음 |
| 재인증이 필요한 중요 작업 | 10분 초과 시 `403 AUTH_REAUTHENTICATION_REQUIRED` |
| 속도 제한 | 429와 `Retry-After: 60`, 계정 실패건수 불변 |
| 서버 오류 | 안전한 `COMMON_INTERNAL_ERROR`, `traceId`, 내부 상세 비노출 |

응답의 HTTP 상태와 오류코드는 `apis.csv`, 필드 구조는 OpenAPI와 일치해야 한다.

## 6. 동시성 테스트

- 로그인실패건수 4인 계정에 비밀번호 오류 두 건을 동시에 보내 최종 6과 비활성 로그 한 건을 확인한다.
- 동일 로그인ID 사용자 등록 두 건에서 완성된 직원·사용자 묶음 한 건만 남는지 확인한다.
- 같은 역할의 메뉴권한을 동시에 변경해 권한 관계와 대상 사용자 보안버전 증가분을 잃지 않는지 확인한다.
- 여러 제한 세션 중 하나에서 최초 등록을 완료한 뒤 나머지 세션이 다음 요청에서 종료되는지 확인한다.
- 계정 비활성과 재활성 처리가 기존 세션 요청과 교차해도 이전 세션이 다시 유효해지지 않는지 확인한다.
- 동일 접속로그에 만료 리스너와 요청 필터가 동시에 TIMEOUT을 기록해도 종료 결과가 한 번만 남는지 확인한다.

## 7. 프론트엔드 테스트

| 대상 | 필수 검증 |
|---|---|
| 로그인 폼 | 중복 제출 방지, 공통 인증 실패 메시지, 성공 후 비밀번호 제거 |
| CSRF | 로그인 전 조회, 로그인·로그아웃·세션 승격 후 새 토큰 조회, 이전 토큰 미사용 |
| 활동 감지 | 키보드·클릭·스크롤·화면 이동만 신호 생성, 폴링·자동조회는 신호 없음 |
| 자동 로그아웃 | 유휴·절대 만료 안내와 로그인 화면 이동 |
| 제한 세션 | 메뉴 미표시, 최초 등록 화면 고정, 허용되지 않은 이동 차단 |
| 권한 변경 | 401 수신 시 캐시·사용자 상태 제거 후 로그인 이동 |
| 중요 작업 | 재인증 요구 시 비밀번호 확인 대화상자, 성공 후 원래 작업 한 번만 재시도 |
| 비밀정보 | 비밀번호·임시 비밀번호를 전역상태·스토리지·URL·분석 이벤트에 저장하지 않음 |

## 8. E2E 핵심 흐름

1. 빈 DB 선행데이터 반영 → 최초 관리자 부트스트랩 → 임시 비밀번호 로그인 → 최초 등록 완료 → 사용자관리 진입
2. 일반 로그인 → 사용자 활동 → 15분 유휴 만료 → 재로그인
3. 로그인 실패 6회 → 계정 비활성 → 관리자 재인증·활성화 → 새 로그인
4. 역할 메뉴권한 변경 → 대상 기존 세션의 다음 요청 종료 → 재로그인 후 새 메뉴 적용
5. 사용자 등록 → 임시 비밀번호 1회 확인 → 제한 세션 → 비밀번호 변경 → 일반 세션 승격
6. 중요 관리자 작업 → 재인증 요구 → 올바른 비밀번호 확인 → 작업 성공과 감사로그
7. 일반 세션에서 활동을 계속해도 8시간 절대 만료

## 10. 필수 통과 기준

- 인증·권한·보안 필수 시나리오는 커버리지 수치와 관계없이 모두 통과해야 한다.
- 단위 테스트는 변경 인증 규칙의 정상·경계·거절 사례를 모두 포함해야 한다.
- API 통합 테스트는 `apis.csv`의 모든 등록 API를 포함해야 한다.
- PostgreSQL 제약·동시성 테스트를 H2 등 다른 DB로 대체하지 않는다.
- 로그인, 최초 등록, 권한 변경, 재인증과 부트스트랩 E2E가 모두 통과해야 인증 구현을 완료로 변경할 수 있다.
- 실패 테스트 출력과 CI 산출물에서 인증비밀이 발견되면 즉시 실패 처리한다.
