# COM-006 공통 애플리케이션 기본 화면 상세설계

## 1. 화면 정보

| 항목 | 내용 |
| --- | --- |
| 설계 상태 | `READY` |
| 화면 ID | `COM-006` |
| 화면명 | 공통 애플리케이션 기본 화면 |
| URL | `/` |
| 기능 패키지 | `FPK-COM-AUTH-001` |
| 관련 기능 | `BFD-02-05-01-02`, `BFD-02-05-01-03` |
| 접근 조건 | 초기 비밀번호 변경을 완료한 일반 인증 세션 |
| v1 역할 | 인증 완료 확인, 사용자·역할·세션 정보 표시, 로그아웃 진입점 |
| 제외 범위 | 대시보드, 알림, 파일, 메모, 업무 데이터 집계, 사용자·역할·메뉴 관리 화면 |

## 2. 화면 목적

로그인과 최초 등록을 완료한 사용자가 일반 인증 세션으로 애플리케이션에
진입했음을 확인할 수 있는 공통 화면을 제공한다. 현재 프런트엔드에 구현된 업무
메뉴가 없을 때 안전한 기본 진입점 역할을 하며, 업무 현황이나 통계를 조회하는
대시보드로 사용하지 않는다.

이 화면은 서버가 로그인 시점에 생성한 사용자·역할·메뉴 스냅샷을 표시하는
읽기 전용 화면이다. 별도의 업무 API를 호출하거나 DB 데이터를 집계하지 않는다.

## 3. 화면 책임과 경계

### 포함

- BMS 브랜드와 현재 사용자 식별정보 표시
- 현재 세션에 포함된 역할 표시
- 일반 세션 유휴시간 기준 표시
- 사용 가능한 구현 메뉴가 없다는 정상 빈 상태 안내
- 수동 로그아웃
- 실제 사용자 활동에 따른 세션 활동 갱신
- 인증 없음·제한 세션·세션 만료에 대한 경로 전환

### 포함하지 않음

- 매출, 프로젝트, 고객, 인력 등 업무 지표
- 알림 개수와 최근 알림
- 최근 파일과 메모
- 구현되지 않은 메뉴의 임시 링크
- 역할·메뉴권한 편집
- 세션 만료시간 연장 버튼
- 사용자 개인정보 상세 표시

향후 업무 화면이 구현되면 공통 헤더와 인증 가드는 공통 애플리케이션 레이아웃으로
분리할 수 있다. v1에서는 `/` 기본 화면 안에 필요한 최소 구조만 둔다.

## 4. 화면 구성

### 4.1 데스크톱

```text
┌──────────────────────────────────────────────────────────────────────┐
│ [B] BMS / Business workspace             관리자(root.admin) [로그아웃]│
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│ WORKSPACE READY                                                      │
│ 로그인되었습니다.                                                    │
│ 인증과 공통 애플리케이션 연결이 완료되었습니다.                      │
│ 현재 구현되어 사용할 수 있는 업무 메뉴가 없습니다.                  │
│                                                                      │
│ ┌──────────────────┬──────────────────┬────────────────────────────┐ │
│ │ 사용자           │ 역할             │ 세션 유휴시간              │ │
│ │ 관리자           │ 시스템관리자     │ 15분                       │ │
│ └──────────────────┴──────────────────┴────────────────────────────┘ │
│                                                                      │
│ ▌업무 화면이 구현되면 권한이 있는 첫 메뉴로 자동 이동합니다.         │
│  이 영역은 대시보드 데이터를 조회하지 않습니다.                     │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

- 상단 헤더는 최대 본문 폭과 정렬을 맞추고 브랜드, 사용자 요약과 로그아웃을
  배치한다.
- 본문 읽기 폭은 과도하게 넓어지지 않도록 최대 폭을 제한한다.
- 사용자·역할·세션 정보는 동일 중요도의 세 카드로 표시한다.
- 카드와 안내 영역은 장식보다 현재 상태 확인을 우선한다.
- 업무 내비게이션, 대시보드 위젯과 빈 차트 영역을 만들지 않는다.

### 4.2 태블릿·모바일

```text
┌──────────────────────────────┐
│ [B] BMS           [로그아웃] │
├──────────────────────────────┤
│ WORKSPACE READY              │
│ 로그인되었습니다.            │
│                              │
│ ┌──────────────────────────┐ │
│ │ 사용자  관리자           │ │
│ ├──────────────────────────┤ │
│ │ 역할    시스템관리자     │ │
│ ├──────────────────────────┤ │
│ │ 세션 유휴시간  15분      │ │
│ └──────────────────────────┘ │
│                              │
│ ▌사용 가능한 메뉴 안내      │
└──────────────────────────────┘
```

- 640px 미만에서는 카드 영역을 한 열로 배치한다.
- 좁은 화면에서는 헤더의 로그인ID를 숨길 수 있으나 사용자명과 로그아웃은
  유지한다.
- 본문 좌우 여백은 최소 20px, 버튼 높이는 최소 44px로 한다.
- 역할명이 길거나 여러 개인 경우 줄바꿈하며 가로 스크롤을 만들지 않는다.
- 320px 너비에서도 핵심 정보와 로그아웃 동작이 손실되지 않아야 한다.

## 5. 화면 진입과 라우팅

1. 애플리케이션 루트에서 `GET /auth/me`로 현재 세션을 확인한다.
2. 조회 중에는 공통 로딩 상태를 표시하고 보호 화면 내용을 먼저 노출하지 않는다.
3. `401 AUTH_AUTHENTICATION_REQUIRED`이면 인증 메모리와 CSRF 토큰을 제거하고
   `/login`으로 이동한다.
4. `200`이고 `passwordChangeRequired=true`이면
   `/account/initial-registration`으로 이동한다.
5. `200`이고 `passwordChangeRequired=false`이면 일반 세션을 구성한다.
6. 응답 메뉴 중 프런트엔드에 구현된 허용 URL이 있으면 `sortOrder`, `menuId`
   순 첫 메뉴로 이동한다.
7. 구현된 허용 메뉴가 없으면 `/`에 머물러 본 화면을 표시한다.

새로고침과 직접 URL 접근도 같은 순서를 사용한다. URL, 브라우저 저장소 또는 이전
클라이언트 상태만으로 인증 여부를 판정하지 않는다.

## 6. 표시 항목

| 영역 | 표시값 | 원본 | 표시 규칙 |
| --- | --- | --- | --- |
| 브랜드 | `BMS`, `Business workspace` | 정적 문구 | 홈 링크로 오인되는 별도 동작을 주지 않음 |
| 사용자명 | `user.displayName` | `CurrentUserResponse` | 빈 값은 정상 응답으로 간주하지 않음 |
| 로그인ID | `user.loginId` | `CurrentUserResponse` | 데스크톱 헤더에만 보조정보로 표시 |
| 역할 | `roles[].roleName` | 세션 스냅샷 | 응답 순서 유지, 쉼표로 구분, 0건은 `없음` |
| 세션 유휴시간 | `idleTimeoutSeconds` | `CurrentUserResponse` | 60으로 나눈 정수 분 단위 표시 |
| 메뉴 안내 | 구현 메뉴 존재 여부 | `menus[]`와 구현 URL 목록 | 0건이면 정상 빈 상태 문구 표시 |

역할명과 메뉴명은 서버가 세션에 저장한 표시정보를 사용한다. 화면 진입 시 역할이나
메뉴 테이블을 별도 조회하지 않는다. 이메일, 휴대전화와 사용자ID는 표시하지 않는다.

## 7. 사용자 동작

| 동작 | 처리 |
| --- | --- |
| 로그아웃 | `API-COM-AUTH-002` 호출 후 인증·CSRF 메모리를 제거하고 `/login`으로 이동 |
| 로그아웃 중 재클릭 | 버튼을 비활성화하고 추가 요청을 만들지 않음 |
| 로그아웃 API 실패 | 로컬 인증정보를 제거하고 `/login`으로 이동하며 보호 화면을 유지하지 않음 |
| 포인터·키보드·스크롤 | 마지막 성공 갱신 후 60초 이상 지난 경우에만 `API-COM-AUTH-004` 호출 |
| 화면 경로 이동 | 일반 사용자 활동으로 간주하되 동일한 60초 제한 적용 |
| 새로고침 | `API-COM-AUTH-003`으로 세션을 다시 판정 |

자동 렌더링, 현재 사용자 조회, CSRF 조회와 백그라운드 API 호출은 사용자 활동으로
간주하지 않는다. 활동 갱신은 절대 세션 만료일시를 연장하지 않는다.

## 8. 화면 상태와 오류

| 상태/오류 | 표시와 동작 |
| --- | --- |
| 세션 확인 중 | 화면 중앙에 “로그인 상태를 확인하고 있습니다.” 진행 상태 표시 |
| 일반 세션·메뉴 없음 | 사용자·역할·유휴시간과 정상 빈 상태 안내 표시 |
| 제한 세션 | 본문을 표시하지 않고 최초 등록 화면으로 이동 |
| 인증 없음 | 본문을 표시하지 않고 로그인 화면으로 이동 |
| 활동 갱신 `401` | 인증정보 제거 후 로그인 화면 이동 |
| 로그아웃 처리 중 | 버튼 비활성화 및 “로그아웃 중...” 표시 |
| `/auth/me` 네트워크 오류 | 안전한 연결 오류, 재시도 버튼과 `traceId`가 있으면 함께 표시 |
| `/auth/me` 서버 오류 | 내부 상세 없이 안전한 메시지와 재시도 방법 표시 |
| 역할 없음 | 오류로 처리하지 않고 역할 카드에 `없음` 표시 |

세션 만료로 로그인 화면에 이동한 경우 “세션이 만료되었습니다” 안내를 한 번
표시한다. 저장 중인 업무 데이터가 없는 화면이므로 복원 기능은 제공하지 않는다.

## 9. API 연결

| 시점 | API ID | 목적 |
| --- | --- | --- |
| 화면 진입·새로고침 | `API-COM-AUTH-003` | 현재 사용자, 역할, 메뉴와 세션 상태 확인 |
| 상태 변경 준비 | `API-COM-AUTH-005` | 로그아웃·활동 갱신용 CSRF 토큰 준비 |
| 사용자 활동 | `API-COM-AUTH-004` | 서버 기준 마지막 사용자 활동시각 갱신 |
| 로그아웃 | `API-COM-AUTH-002` | 서버 세션과 접속로그 종료 |

공통 클라이언트는 `/backend-api` 동일 출처 프록시를 사용한다. 화면 컴포넌트에서
직접 `fetch`하거나 세션 쿠키를 읽지 않는다.

## 10. 접근성과 보안

- 문서 언어는 한국어, 페이지 제목은 “공통 애플리케이션 | BMS”로 제공한다.
- 페이지에는 하나의 명확한 `h1`을 제공한다.
- 로딩 상태는 `role="status"`, 오류 안내는 `role="alert"`로 전달한다.
- 로그아웃 버튼은 키보드로 접근·실행할 수 있고 처리 중 상태를 텍스트로 알린다.
- 색상만으로 현재 상태, 카드 구분과 오류를 전달하지 않는다.
- 포커스 표시를 제거하지 않고 키보드 순서는 브랜드 → 사용자 정보 → 로그아웃 →
  본문 순서와 충돌하지 않게 한다.
- 사용자명과 로그인ID를 URL, 브라우저 저장소와 분석 이벤트에 기록하지 않는다.
- 세션ID, 쿠키, CSRF 토큰과 전체 인증 응답을 콘솔이나 오류 화면에 출력하지 않는다.
- 역할과 메뉴는 표시 용도로만 사용하며 서버 권한 검사를 대체하지 않는다.
- 응답과 화면은 캐시하지 않고 새로고침 시 서버 세션을 다시 확인한다.

## 11. 인수조건

- 미인증 사용자가 `/`에 접근하면 보호 내용을 보지 못하고 `/login`으로 이동한다.
- 제한 세션 사용자가 `/`에 접근하면 `/account/initial-registration`으로 이동한다.
- 일반 세션 사용자는 사용자명, 로그인ID, 역할과 유휴시간을 확인할 수 있다.
- 구현된 허용 메뉴가 없으면 오류나 대시보드 대신 정상 안내 화면을 표시한다.
- 화면 진입 시 현재 사용자 조회 외 업무·집계·알림·파일·메모 API를 호출하지 않는다.
- 사용자 활동 이벤트가 60초 안에 중복 활동 요청을 만들지 않는다.
- 활동 갱신에서 `401`을 받으면 인증정보를 제거하고 로그인 화면으로 이동한다.
- 로그아웃 요청이 성공하거나 실패해도 보호 화면을 유지하지 않는다.
- 320px 너비에서 내용 손실과 가로 스크롤이 없다.
- 키보드만으로 화면 내용을 확인하고 로그아웃할 수 있다.
- 비밀번호, 세션ID, 쿠키와 CSRF 토큰이 화면·URL·브라우저 저장소·로그에 노출되지 않는다.

## 12. 구현 파일 기준

초안 기준 구현 경계는 다음과 같다.

```text
frontend/
├── app/
│   └── page.tsx
├── features/
│   ├── auth/
│   │   ├── AuthGate.tsx
│   │   ├── AuthProvider.tsx
│   │   ├── HomeClient.tsx
│   │   └── LogoutButton.tsx
│   └── session/
│       └── SessionActivityTracker.tsx
└── shared/
    ├── api/
    │   ├── api-client.ts
    │   └── auth-api.ts
    └── types/
        └── auth.ts
```

- `page.tsx`는 화면 조합과 메타데이터만 담당한다.
- 인증 상태와 세션 스냅샷은 `AuthProvider`가 소유한다.
- HTTP 요청은 `shared/api` 밖에서 직접 수행하지 않는다.
- 대시보드·알림·파일·메모 파일이나 임시 데이터를 추가하지 않는다.

## 13. 구현 확정사항

- `COM-006`을 화면목록의 공통 애플리케이션 기본 화면 ID로 사용한다.
- v1에서는 현재 세션의 역할명을 모두 표시하고 역할이 없으면 `없음`으로 표시한다.
- 세션 만료 안내는 브라우저 저장소나 URL을 사용하지 않고 `AuthProvider` 메모리에
  보관한 뒤 로그인 화면에서 한 번 소비한다.
- 보호 화면의 세션 확인 오류는 안전한 메시지와 추적ID를 표시하고 사용자가
  `/auth/me` 조회를 다시 시도할 수 있게 한다.
- 향후 구현된 허용 업무 메뉴가 생기면 현재 계약대로 첫 허용 메뉴로 자동 이동한다.

## 14. 구현 참조
