# COM-001 로그인 화면 상세설계

## 1. 화면 정보

| 항목 | 내용 |
| --- | --- |
| 설계 상태 | `READY` |
| 화면 ID | `COM-001` |
| URL | `/login` |
| 기능 패키지 | `FPK-COM-AUTH-001` |
| 관련 기능 | `BFD-02-05-01-01`, `BFD-02-05-01-02`, `BFD-02-05-01-03` |
| 접근 조건 | 비인증 사용자 접근 가능 |
| 제외 범위 | 비밀번호 찾기, 회원가입, SSO, MFA, 대시보드 |

## 2. 화면 목적

로그인ID와 비밀번호를 안전하게 입력받아 서버 세션을 생성한다. 이미 유효한 세션이 있으면 세션 종류에 맞는 화면으로 이동하고, 실패 원인이나 계정 존재 여부를 외부에 노출하지 않는다.

## 3. 화면 구성

### 데스크톱

```text
┌──────────────────────────┬───────────────────────────────────┐
│                          │ BMS                               │
│ 업무를 연결하고          │ Business Management System        │
│ 운영을 단순하게          │                                   │
│                          │ 로그인ID                          │
│ 공통 브랜드 영역         │ [                               ] │
│                          │ 비밀번호                          │
│                          │ [••••••••••••             ] [표시]│
│                          │                                   │
│                          │ [오류 또는 처리 안내]             │
│                          │ [             로그인             ]│
└──────────────────────────┴───────────────────────────────────┘
```

- 1024px 이상에서는 브랜드 영역과 로그인 폼을 1:1에 가까운 두 열로 배치한다.
- 폼 영역의 읽기 폭은 420px 이하로 제한한다.
- 장식보다 입력과 오류 확인을 우선하며 업무 데이터·통계·알림을 표시하지 않는다.

### 모바일

- 768px 미만에서는 한 열로 전환하고 브랜드 설명을 축약한다.
- 본문 좌우 여백은 최소 20px, 입력과 버튼 높이는 최소 44px로 한다.
- 세로 640px 환경에서도 로그인 버튼까지 자연스럽게 스크롤할 수 있어야 한다.
- 가로 스크롤을 만들지 않는다.

## 4. 화면 진입

1. `GET /auth/me`로 기존 세션을 확인한다.
2. `200`이고 `passwordChangeRequired=false`이면 `/`로 이동한다.
3. `200`이고 `passwordChangeRequired=true`이면 `/account/initial-registration`으로 이동한다.
4. `401 AUTH_AUTHENTICATION_REQUIRED`이면 로그인 폼을 유지한다.
5. 세션 확인과 병렬로 요청하지 않고, 미인증이 확인된 뒤 `GET /auth/csrf`로 로그인용 토큰을 준비한다.
6. 네트워크 오류로 세션 확인에 실패하면 폼은 표시하되 연결 오류를 안내하고 제출 시 다시 시도한다.

초기 포커스는 세션 확인이 끝난 뒤 로그인ID에 둔다. 브라우저 자동완성으로 이미 값이 있는 경우 포커스를 강제로 덮어쓰지 않는다.

## 5. 입력 필드

| 필드 | HTML/자동완성 | 필수 | 클라이언트 검증 | 제출 처리 |
| --- | --- | :---: | --- | --- |
| 로그인ID | `text`, `username` | Y | 앞뒤 공백 제거 후 1~100자 | 앞뒤 공백 제거 및 영문 소문자 변환 |
| 비밀번호 | `password`, `current-password` | Y | 공백 포함 원문 1~256자 | 변형 없이 전송 |

로그인 화면에서는 신규 비밀번호 복잡도 정책을 검사하지 않는다. 비밀번호의 앞뒤 공백도 자격 증명의 일부일 수 있으므로 제거하지 않는다.

## 6. 버튼과 키보드 동작

| 동작 | 처리 |
| --- | --- |
| 로그인 | 클라이언트 검증 후 `API-COM-AUTH-001` 호출 |
| Enter | 폼 제출과 동일하게 처리하며 제출 중이면 무시 |
| 비밀번호 표시 | 현재 비밀번호 필드만 표시·숨김하고 버튼의 접근성 이름을 “비밀번호 표시/숨기기”로 변경 |

제출 중에는 입력, 표시 버튼과 로그인 버튼을 비활성화한다. 요청이 종료되기 전 추가 클릭과 Enter 입력은 새 요청을 만들지 않는다.

## 7. 로그인 성공 처리

1. 비밀번호 값을 즉시 비운다.
2. 로그인 전 CSRF 토큰을 폐기한다.
3. 응답의 `passwordChangeRequired`를 확인한다.
4. 제한 세션이면 `/account/initial-registration`으로 이동하고 해당 화면에서 CSRF 토큰을 새로 받는다.
5. 일반 세션이면 현재 사용자 정보를 인증 상태에 반영한다.
6. 구현된 허용 메뉴가 있으면 `sortOrder`, `menuId` 순 첫 메뉴로 이동한다.
7. 구현된 허용 메뉴가 없으면 `/`의 공통 애플리케이션 기본 화면으로 이동한다.

공통 기본 화면은 로그인 완료와 사용자 정보를 안내하는 진입점이며 대시보드 기능을 포함하지 않는다.

## 8. 화면 상태와 오류

| 상태/오류 | 표시와 동작 |
| --- | --- |
| 세션 확인 중 | 폼 자리에 로딩 상태 표시, 제출 불가 |
| 초기 | 로그인ID와 비밀번호 입력 가능 |
| 필수·길이 오류 | 필드 인접 메시지, 첫 오류 필드로 포커스 |
| 제출 중 | 입력·버튼 비활성화, 로그인 버튼에 처리 중 표시 |
| `AUTH_LOGIN_FAILED` | “로그인 정보를 확인해 주세요.”를 폼 상단에 표시하고 비밀번호 제거 |
| `AUTH_TOO_MANY_ATTEMPTS` | `Retry-After` 동안 제출 비활성화, 남은 초 표시, 비밀번호 제거 |
| `AUTH_CSRF_INVALID` | CSRF 토큰을 새로 받아 동일 제출을 한 번만 재시도 |
| `COMMON_VALIDATION_FAILED` | `fieldErrors`를 입력과 연결하고 비밀번호 제거 |
| 네트워크 오류 | “서버에 연결할 수 없습니다. 잠시 후 다시 시도해 주세요.” 표시 |
| 서버 오류 | 안전한 메시지와 `traceId`, 다시 시도 안내 표시 |

계정 비활성 여부, 로그인ID 존재 여부, 실패 횟수, 내부 실패사유는 표시하지 않는다. 오류 이후 로그인ID는 유지하고 비밀번호는 항상 비운다.

## 9. API 연결

| 순서 | API ID | 목적 |
| ---: | --- | --- |
| 1 | `API-COM-AUTH-003` | 기존 세션 확인 |
| 2 | `API-COM-AUTH-005` | 로그인용 CSRF 토큰 준비 |
| 3 | `API-COM-AUTH-001` | 로그인과 세션 생성 |

요청:

```json
{
  "loginId": "root.admin",
  "password": "current-password-value"
}
```

성공 응답의 필드와 오류코드는 OpenAPI와 `apis.csv`를 원본으로 한다.

## 10. 접근성과 보안

- 문서 언어는 한국어로 설정하고 페이지 제목은 “로그인 | BMS”로 제공한다.
- 모든 입력에 화면에 보이는 레이블을 연결한다.
- 오류는 `aria-describedby`, 폼 공통 오류는 `role="alert"`로 연결한다.
- 제출 중 상태는 텍스트로도 알리고 색상·회전 아이콘에만 의존하지 않는다.
- 키보드 포커스 표시를 제거하지 않는다.
- 로그인ID를 URL query, 브라우저 저장소와 분석 이벤트에 기록하지 않는다.
- 비밀번호를 폼 바깥 전역 상태, 브라우저 저장소, 콘솔과 오류 객체에 보관하지 않는다.
- 세션 쿠키, CSRF 토큰과 응답 본문을 로그에 출력하지 않는다.
- 브라우저 뒤로 가기와 새로고침 후 비밀번호 값을 복원하지 않는다.

## 11. 인수조건

- 미인증, 일반 세션과 제한 세션 진입이 각각 올바른 화면으로 분기된다.
- 필수값 없이 제출하면 API를 호출하지 않고 첫 오류 필드로 포커스가 이동한다.
- 제출 중 클릭과 Enter가 중복 로그인 요청을 만들지 않는다.
- 인증 실패 원인과 무관하게 같은 메시지가 표시되고 비밀번호가 지워진다.
- 속도 제한 동안 남은 시간이 표시되고 만료 전 재제출할 수 없다.
- 로그인 성공 후 이전 CSRF 토큰을 재사용하지 않는다.
- 제한 세션은 `/account/initial-registration`, 일반 세션은 구현 메뉴 또는 `/`로 이동한다.
- 키보드만으로 입력, 비밀번호 표시, 제출과 오류 확인이 가능하다.
- 320px 너비에서 내용 손실과 가로 스크롤이 없다.

## 12. 구현 참조

공통 프록시, API 클라이언트, 상태 소유권, 파일 배치와 검증 기준은 상위 패키지의 `frontend-implementation-contract.md`를 따른다.
