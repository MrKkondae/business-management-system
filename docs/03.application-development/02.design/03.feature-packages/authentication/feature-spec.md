# 인증 기능 명세 초안

## 1. 문서 정보

| 항목 | 내용 |
| --- | --- |
| 기능 패키지 | `FPK-COM-AUTH-001` |
| 백로그 | `PB-001` 로그인 |
| 요구사항 | `REQ-COM-005`, `NFR-SEC-001`, `NFR-SEC-002` |
| 화면 | `COM-001` |
| 상태 | 초안 (`DESIGN_IN_PROGRESS`) |

이 문서는 로그인, 로그아웃, 현재 사용자 조회와 세션 활동 갱신의 구현 경계를 정의한다. 사용자 등록·비밀번호 변경과 역할·메뉴 기준정보 관리는 각각 사용자 관리와 역할·권한 관리 패키지에서 다룬다.

## 2. 사용자 가치와 범위

사용자는 로그인ID와 비밀번호로 인증하고, 자신에게 허용된 메뉴만 확인하며, 직접 로그아웃하거나 15분 동안 활동이 없으면 자동 로그아웃된다.

### 포함

- 로그인 성공 시 서버 세션과 역할·메뉴권한 스냅샷 생성
- 로그인 성공·실패 횟수와 계정 상태 처리
- 로그인 성공·실패, 수동 로그아웃과 자동 만료 접속로그
- 현재 사용자, 역할과 접근 가능 메뉴 조회
- 사용자 활동에 따른 세션 유휴시간 갱신
- 로그인 전 CSRF 토큰 발급
- 중요 관리자 작업 전 현재 비밀번호 재인증

### 제외

- 사용자 등록·수정, 비밀번호 초기화·변경
- 역할·메뉴와 역할별 메뉴권한 기준정보 변경
- SSO, OAuth, 공개 회원가입과 비밀번호 찾기
- 동시 로그인 제한과 관리자 세션 강제 종료 화면

## 3. 기능과 API 연결

| 기능 ID | 기능 | API |
| --- | --- | --- |
| `BFD-02-05-01-01` | 로그인 | `API-COM-AUTH-001`, `API-COM-AUTH-005` |
| `BFD-02-05-01-02` | 로그아웃 | `API-COM-AUTH-002` |
| `BFD-02-05-01-03` | 현재 사용자 조회 | `API-COM-AUTH-003`, `API-COM-AUTH-004` |

`API-COM-AUTH-004`와 `API-COM-AUTH-005`는 각각 세션 활동 갱신과 로그인 전 CSRF 준비를 위한 보조 API이며 별도 사용자 업무기능으로 세분하지 않는다.

| `AUTH-BR-010` | 초기 비밀번호 변경이 필요한 사용자는 10분 유휴·30분 절대 만료 제한 세션을 발급하고 일반 업무 메뉴를 제공하지 않는다. 초기 비밀번호 만료일시가 더 빠르면 그 시각을 제한 세션 만료 상한으로 사용한다. |
| `AUTH-BR-011` | 유효한 활동 API만 `lastUserActivityAt`을 갱신한다. 업무 API, 현재 사용자 조회, 폴링과 자동 갱신은 컨테이너 접근시각과 관계없이 인증 세션 유휴시간을 연장하지 않는다. |
| `AUTH-BR-012` | 로그인ID·IP 조합은 1분에 5회, IP 전체는 1분에 30회까지 허용하고 초과 요청은 60초 동안 `429`로 거절한다. |
| `AUTH-BR-013` | 속도 제한으로 거절한 요청은 비밀번호를 검증하지 않고 계정 로그인실패건수를 증가시키지 않는다. |

| `AUTH-BR-015` | v1.0은 단일 백엔드 인스턴스의 기본 `HttpSession`을 사용하며 서버 재시작 시 기존 세션 소멸을 허용한다. |
| `AUTH-BR-016` | 모든 보호 API는 현재 계정의 활성·삭제 상태를 확인한다. 비활성·삭제 계정의 기존 세션은 다음 보호 요청에서 차단·만료하며, 요청이 없으면 마지막 사용자 활동부터 최대 15분 안에 만료한다. |
| `AUTH-BR-017` | 제한 세션은 로그인 당시 `PWD_CHG_DTM`을 자격 증명 기준시각으로 저장하고 모든 허용 요청에서 현재 값과 비교한다. 임시 비밀번호 재발급 또는 다른 제한 세션의 최초 등록 완료로 값이나 초기화필요여부가 달라지면 세션을 만료한다. |
| `AUTH-BR-018` | 최초 등록 완료 시 새 비밀번호 정책과 현재 임시 비밀번호 불일치를 검증하고 비밀번호 해시, 초기화필요여부와 임시 비밀번호 만료일시를 한 트랜잭션에서 변경한 뒤 세션ID 교체 후 새 권한 스냅샷의 일반 세션으로 승격한다. |
| `AUTH-BR-019` | 일반 세션은 15분 유휴시간과 로그인 또는 제한 세션 승격부터 8시간 절대 유효시간을 함께 적용하며 활동 API는 절대 유효시간을 연장하지 않는다. |
| `AUTH-BR-020` | 모든 보호 API는 현재 계정의 `SEC_VER`와 세션 `securityVersion`을 비교하고 다르면 업무 처리 전에 세션을 만료한다. |
| `AUTH-BR-021` | 로그인ID가 없어도 더미 Argon2id 해시를 검증하고 실제 비밀번호는 Argon2id 최소 메모리 19 MiB·반복 2회·병렬도 1로 검증한다. |
| `AUTH-BR-022` | 중요 관리자 작업은 로그인 또는 `POST /auth/reauthenticate` 성공 후 10분 안에서만 허용한다. |
| `AUTH-BR-023` | 재인증은 현재 비밀번호와 CSRF를 검증하고 세션별 분당 5회로 제한한다. 비밀번호 불일치는 `AUTH_REAUTHENTICATION_FAILED`로 응답하고 계정 로그인실패건수를 증가시키지 않는다. |
| `AUTH-BR-024` | CSRF 검증 실패는 `403 AUTH_CSRF_INVALID`로 통일하고 로그인 성공·로그아웃·최초 등록 완료 후 토큰을 교체한다. |

## 5. 처리 흐름

### 5.1 로그인

1. 클라이언트가 `GET /auth/csrf`로 로그인 요청용 CSRF 토큰을 준비한다.
2. 로그인ID의 앞뒤 공백을 제거하고 영문자를 소문자로 변환한 뒤 필수값을 검증하며 신뢰 프록시 기준 접속 IP를 확인한다.
3. 메모리 기반 속도 제한기에서 로그인ID·IP 조합과 IP 전체의 1분 요청 수를 확인한다.
4. 제한을 초과하면 자격 증명을 검증하지 않고 `Retry-After: 60`과 `429`를 반환한다.
5. 삭제되지 않은 사용자를 정규화 로그인ID로 조회한다. 사용자가 없으면 고정된 더미 Argon2id 해시를 검증한다.
6. 활성 상태와 비밀번호를 검증한다.
7. 실패하면 등록 사용자에 한해 실패건수를 원자적으로 증가시키고, 증가 결과가 6이면 동일 트랜잭션에서 계정을 비활성화한다.
8. 실패 접속로그를 저장하고 사용자 존재 여부와 무관하게 동일한 401 응답을 반환한다.
9. 성공하면 실패건수를 0으로 만들고 로그인ID·IP 조합의 속도 제한 카운터를 초기화하며 마지막 로그인 일시를 갱신한다.
10. 활성 역할과 접근 가능 메뉴를 조회하여 현재 `SEC_VER`, `absoluteExpiresAt=로그인시각+8시간`, `reauthenticatedAt=로그인시각`을 포함한 권한 스냅샷을 만든다.
11. 세션 식별자를 교체하고 성공 접속로그ID를 세션에 연결한다.
12. 사용자 요약정보와 권한 스냅샷을 반환한다.

로그인 실패건수 증가에는 조건부 갱신 또는 행 잠금을 사용하여 동시 실패 요청이 증가분을 잃지 않게 한다. 실패건수 변경, 6번째 실패 비활성화와 접속로그 저장은 한 로그인 시도 단위의 트랜잭션으로 처리한다.

### 5.2 로그아웃

1. 세션에 연결된 접속로그가 있으면 로그아웃일시와 `MANUAL`을 기록한다.
2. 서버 세션을 무효화하고 세션·CSRF 쿠키를 만료한다.
3. 처리가 끝나면 `204 No Content`를 반환한다. 세션이 없거나 만료되었으면 공통 인증 필요 오류를 반환한다.

### 5.3 현재 사용자 조회

세션의 사용자ID, 표시명, 역할과 접근 가능 메뉴 스냅샷을 반환한다. 비밀번호 초기화가 필요한 제한 세션이면 `passwordChangeRequired=true`와 빈 메뉴 목록을 반환한다. 세션이 없거나 만료되면 `AUTH_AUTHENTICATION_REQUIRED`를 반환한다.

### 5.4 활동과 자동 만료

- 클라이언트는 실제 사용자 활동이 있으면서 직전 신호 후 1분 이상 지난 경우에만 활동 API를 호출한다.
- 서버는 계정의 활성·삭제 상태와 세션의 유효성을 확인한 뒤 별도 `lastUserActivityAt`을 갱신하고 `204`를 반환한다.
- 일반 세션은 15분 유휴시간과 로그인 성공부터 8시간의 절대 유효시간을 적용한다. 제한 세션은 10분 유휴시간과 로그인 성공부터 30분의 절대 유효시간을 적용하며 활동 API는 어느 세션의 절대 유효시간도 연장하지 않는다.
- 활동 API 외의 업무 API, 현재 사용자 조회, CSRF 조회와 백그라운드 요청은 `lastUserActivityAt`을 갱신하지 않는다.
- 15분 만료 시 접속로그에 `TIMEOUT`을 한 번 기록한다. 중복 만료 이벤트는 같은 접속로그를 다시 변경하지 않는다.

### 5.5 초기 비밀번호 제한 세션

1. 초기 비밀번호가 맞더라도 `TEMP_PWD_EXPIRE_DTM`이 지났으면 제한 세션을 발급하지 않고 로그인실패건수를 증가시키지 않는다. 외부에는 일반 `401 AUTH_LOGIN_FAILED`, 내부 접속로그에는 `TEMP_PWD_EXPIRED`를 기록한다.
2. 제한 세션에는 `lastUserActivityAt`, `absoluteExpiresAt=min(로그인시각+30분, TEMP_PWD_EXPIRE_DTM)`과 로그인 당시 `PWD_CHG_DTM`을 저장한다.
3. 제한 세션은 `GET /auth/csrf`, `GET /auth/me`, `POST /auth/activity`, `POST /users/me/initial-registration`, `POST /auth/logout`만 호출할 수 있다.
4. 각 허용 요청에서 계정 활성·삭제 상태, `PWD_INIT_REQ_YN`, 임시 비밀번호 만료와 로그인 당시 `PWD_CHG_DTM` 일치를 확인한다.
5. 최초 등록 완료 시 새 비밀번호 정책과 현재 임시 비밀번호 불일치를 검증하고 새 비밀번호 해시, 선택 개인정보, `PWD_INIT_REQ_YN='N'`, `TEMP_PWD_EXPIRE_DTM=NULL`, `SEC_VER=SEC_VER+1`을 하나의 트랜잭션으로 저장한다.
6. 커밋 후 세션ID와 CSRF 토큰을 교체하고 활성 역할·메뉴 표시정보와 변경된 보안버전을 다시 조회한다. 일반 세션의 `lastUserActivityAt`·`reauthenticatedAt`은 완료시각, `absoluteExpiresAt`은 완료시각+8시간으로 설정한다.
7. 등록 완료 전 브라우저 종료나 네트워크 장애가 발생하면 임시 비밀번호 유효기간 안에서 다시 로그인할 수 있다. 완료 또는 임시 비밀번호 재발급 후에는 이전 임시 비밀번호를 사용할 수 없다.

| `TB_SYS_ACCESS_LOG` | 생성·수정 | 로그인 시도와 로그아웃 기록 |
| `TB_SYS_LOG` | 생성 | 6번째 실패 계정 비활성화 사건 기록 |

로그인 실패 내부 사유는 `USER_NOT_FOUND`, `BAD_CREDENTIALS`, `ACCOUNT_INACTIVE`로 접속로그에 구분할 수 있지만 외부 응답에는 노출하지 않는다. 미등록 로그인ID는 원문 대신 서버 비밀키를 사용하는 일방향 해시 또는 동등한 비식별 값만 기록한다.

## 7. 세션과 보안

- 세션 쿠키: `HttpOnly`, 운영 `Secure`, `SameSite=Lax`
- 세션 쿠키명 `BMSSESSION`, `Path=/`, Domain·Max-Age·Expires 미지정
- 인증 성공 시 세션 고정 공격 방지를 위해 세션ID 교체
- 로그인·로그아웃·재인증을 포함한 상태 변경 API는 CSRF 헤더 검증
- 로그인 전 CSRF API는 비인증 호출을 허용하되 인증 세션을 만들지 않음
- 인증 응답과 서버 로그에서 계정 존재 여부를 추론할 수 있는 메시지 차이를 제거
- 접속 IP는 신뢰 프록시 목록을 거쳐 확인한 값만 사용
- 로그인ID·IP 조합은 1분에 5회, IP 전체는 1분에 30회로 제한하고 60초 후 다시 허용
- 속도 제한 키는 정규화 로그인ID 원문 대신 서버 내부 비가역 키와 접속 IP 조합 사용
- 속도 제한기는 `LoginAttemptLimiter` 인터페이스로 분리하고 v1.0 구현은 단일 인스턴스 메모리 저장소 사용

### 7.1 세션 저장 정책

- v1.0 개발·운영환경은 단일 백엔드 인스턴스와 서블릿 컨테이너 기본 `HttpSession`을 사용한다.
- 서버 재시작·재배포 시 기존 세션이 소멸되어 모든 사용자가 다시 로그인하는 것을 허용한다.
- 일반 세션에는 사용자ID, 로그인ID, 표시명, 역할ID·역할명 목록, 메뉴ID·상위메뉴ID·메뉴명·메뉴URL·표시순서 목록, 접속로그ID, 비밀번호변경필요여부, `securityVersion`, `lastUserActivityAt`, `absoluteExpiresAt`과 `reauthenticatedAt`을 불변 DTO로 저장한다.
- 제한 세션에는 위 사용자 식별정보와 접속로그ID 외에 빈 역할·메뉴, 비밀번호변경필요여부, `lastUserActivityAt`, `absoluteExpiresAt`과 로그인 당시 `PWD_CHG_DTM`을 저장한다.
- JPA 엔터티, 비밀번호·해시, 세션 쿠키값, CSRF 토큰과 불필요한 개인정보를 세션 속성으로 저장하지 않는다.
- 세션 생성·조회·종료는 `AuthSessionService` 경계에 모으고 업무 코드가 컨테이너 구현체에 직접 의존하지 않게 한다.
- 모든 보호 API 요청은 DB에서 현재 계정의 활성·삭제 상태와 `SEC_VER`를 확인한다. 비활성·삭제 또는 보안버전 불일치이면 요청을 차단하고 현재 세션을 만료 처리하되 사용자별 세션 목록은 구현하지 않는다.
- 정상적인 15분 만료는 세션 만료 리스너로 접속로그를 `TIMEOUT` 처리한다.
- 단일 인스턴스 기동 시 이전 실행에서 로그아웃일시가 없는 성공 접속로그를 기동시각 기준 `TIMEOUT`으로 보정한다.
- 다중 인스턴스, 무중단 배포 또는 재시작 후 세션 유지가 필요해질 때 별도 ADR로 공유 세션 저장소를 검토한다.

## 8. 오류 계약

외부 오류코드 정의는 `../../02.catalog/error-codes.csv`, 인증 API별 오류 연결은 `../../02.catalog/apis.csv`를 원본으로 한다. 오류 응답은 공통 `Problem` 구조와 `traceId`를 사용하며 내부 로그인·재인증 실패 원인을 노출하지 않는다.

## 9. 완료 조건

- 기능·화면·API·OpenAPI와 인수 시나리오의 ID가 일치한다.
- 1~5회 실패, 6번째 실패, 성공 초기화와 동시 실패 테스트가 정의된다.
- 로그인 응답이 사용자 존재 여부와 내부 실패 원인을 노출하지 않는다.
- 세션 고정, CSRF, 15분 만료와 권한 스냅샷 테스트가 정의된다.
- 접속로그 및 계정 비활성 시스템로그의 생성·마스킹을 검증한다.
- 로그인ID·IP와 IP 전체 속도 제한, `Retry-After`와 실패건수 미증가를 검증한다.
- 서버 재기동 시 메모리 세션 소멸과 미종료 접속로그 보정을 검증한다.
- 로그인ID가 소문자로 정규화되어 저장·조회되고 대소문자만 다른 입력이 같은 계정으로 처리됨을 검증한다.
- 활동 API만 `lastUserActivityAt`을 갱신하고 업무 API와 백그라운드 요청은 유휴시간을 연장하지 않음을 검증한다.
- 계정 비활성화 후 기존 세션의 다음 보호 API 요청이 차단됨을 검증한다.
- 현재 사용자 응답이 세션에 저장한 역할명과 메뉴 표시정보 스냅샷을 사용함을 검증한다.
- 초기 비밀번호 24시간 만료, 제한 세션 10분 유휴·30분 절대 만료와 활동 API의 절대 만료 미연장을 검증한다.
- 제한 세션 API 허용목록, 임시 비밀번호 재발급 후 기존 세션 차단과 최초 등록 완료 후 일반 세션 승격을 검증한다.
- 일반 세션이 활동 중에도 8시간에 절대 만료됨을 검증한다.
- 역할·메뉴권한 변경으로 보안버전이 증가하고 기존 세션의 다음 보호 요청이 차단됨을 검증한다.
- 중요 작업 10분 재인증, 분당 5회 제한과 재인증 실패 시 계정 실패건수 미증가를 검증한다.
- Argon2id 파라미터, 미등록 사용자 더미 해시와 세션·CSRF·보안 헤더 계약을 검증한다.

## 10. 확정 결정

- 로그인 속도 제한은 로그인ID·IP 조합 5회/분, IP 전체 30회/분이며 60초 동안 요청을 거절한다.
- 최초 비밀번호 변경과 선택 개인정보 완성 화면은 사용자 관리 패키지가 소유하는 `/account/initial-registration`을 사용한다.
- v1.0은 단일 인스턴스 기본 `HttpSession`을 사용하고 공유 세션 저장소는 도입하지 않는다.
- 현재 확장 예상인 자사직원·외주인력의 개인 학력·경력 입력과 약 50명 규모는 공유 세션 저장소 도입 조건으로 보지 않는다.
- 일반 세션은 별도 `lastUserActivityAt` 기준 15분이며 유효한 활동 API만 이 값을 갱신한다.
- 계정 비활성화 시 전체 세션 즉시 종료는 구현하지 않고 다음 보호 API 요청에서 차단하며, 요청이 없으면 최대 15분 안에 만료한다.
- 로그인ID는 앞뒤 공백 제거 후 소문자로 변환하여 소문자 값만 저장·조회한다.
- 세션에는 역할명과 메뉴명·상위메뉴ID·URL·표시순서를 포함한 표시정보 스냅샷을 저장한다.
- 최초 등록 전용 임시 비밀번호는 24시간 유효하고 완료 전 재로그인을 허용한다. 제한 세션은 유휴 10분·절대 30분이며 완료 시 세션ID 교체 후 일반 세션으로 승격한다.
- 일반 세션은 유휴 15분·절대 8시간이며 권한 변경은 `SEC_VER` 증가로 기존 세션을 다음 요청에서 종료한다.
- 중요 관리자 작업은 최근 10분 비밀번호 재인증을 요구하고 시스템관리자 MFA는 v1.0 이후 우선 보안과제로 분리한다.
- 세부 기술 계약은 `../../01.common/09.security-implementation-contract.md`를 따른다.
