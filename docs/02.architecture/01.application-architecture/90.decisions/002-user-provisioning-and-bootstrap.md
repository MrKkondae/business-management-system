# ADR-002 사용자 프로비저닝과 최초 관리자 부트스트랩

## 상태

승인

## 일자

2026-07-22

## 배경

현재 회사는 직원 기준정보를 별도 시스템이나 파일로 관리하지 않는다. 따라서 사용자 계정을 만들기 전에 가져올 직원 원본이 없고, 사용자관리보다 인력관리 전체 기능을 먼저 구현하는 것도 첫 번째 수직 단위의 범위를 불필요하게 키운다.

동시에 사람 사용자 계정은 내부 직원과만 연결하고 외주인력 계정은 만들지 않는 기존 정책과, 시스템관리 모듈이 인력관리 데이터베이스를 직접 변경하지 않는 모듈 소유권을 유지해야 한다. 최초 배포 시에는 사용자를 등록할 관리자가 아직 존재하지 않는 부트스트랩 문제도 해결해야 한다.

## 결정

- 공개 회원가입은 제공하지 않는다.
- 빈 시스템의 최초 시스템관리자는 배포 작업자가 실행하는 1회성 부트스트랩 절차로 생성한다.
- 부트스트랩은 DB 직접 입력이나 공개 API가 아니라 일반 HTTP 포트를 열지 않는 애플리케이션 전용 실행 모드로 제공한다.
- 부트스트랩은 기본 조직, 관리자 직원정보, 사용자 계정과 시스템관리자 역할 관계를 하나의 트랜잭션에서 생성한다.
- 역할, 메뉴, 역할-메뉴 관계와 공통코드는 검증된 CSV를 원본으로 신규 Flyway 마이그레이션을 통해 먼저 반영한다.
- 최초 시스템관리자 역할에는 운영 DB에 등록된 모든 유효 메뉴를 명시적으로 부여하며 권한 검사 우회를 사용하지 않는다.
- 초기 비밀번호는 배포환경의 일회성 비밀값으로 전달하고 소스코드, Flyway 파일, 영속 설정과 로그에 남기지 않는다.
- 이후 사용자는 관리자가 `/system/users`에서 사원번호, 성명, 조직, 로그인ID와 초기 역할을 직접 입력하여 생성한다.
- 사용자관리 유스케이스는 인력관리 모듈의 공개 등록 인터페이스를 호출한다. 인력관리 모듈이 업무대상·인력·직원을 생성하고 시스템관리 모듈이 반환된 직원ID로 사용자와 역할 관계를 생성한다.
- 사용자관리 1차 등록은 신규 내부 직원과 사용자 계정을 함께 생성한다. 이미 존재하는 사원번호는 중복으로 거부하고 기존 직원에 사용자 계정을 연결하는 기능은 후속 단위로 분리한다.
- 사람의 성명·이메일·휴대전화 기준 원본은 `TB_RES_MST`로 한다. `TB_SYS_USER`의 동명 컬럼은 인증·표시용 사본이며 사용자 생성, 최초 등록과 이후 직원정보 변경 시 인력관리 공개 인터페이스를 통해 같은 트랜잭션으로 동기화한다.
- 사용자관리 1차 등록에서는 사원번호·성명·조직과 선택 연락처만 입력한다. 입사일자와 직위는 `NULL`, 인력구분은 `EMPLOYEE`, 재직상태는 `EMPLOYED`로 생성하고 상세값은 후속 직원관리에서 완성한다.
- 위 생성 작업은 하나의 트랜잭션으로 묶고 일부만 성공한 데이터를 남기지 않는다.
- 사용자는 24시간 유효한 최초 등록 전용 임시 비밀번호로 제한 세션에 로그인한 뒤 새 비밀번호와 선택 개인정보를 입력하여 등록을 완성한다. 등록을 완료하기 전에는 유효기간 안에서 제한 세션에 다시 로그인할 수 있고, 완료 또는 재발급 즉시 이전 임시 비밀번호는 사용할 수 없다.
- 직원 엑셀 일괄등록은 v1.0에서 제외하고 실제 운영량과 반복 입력 비용이 확인된 후 별도 백로그로 검토한다.

## 영향

- 첫 번째 사용자관리 수직 단위에 인력관리의 최소 직원 등록 인터페이스와 공통 업무대상 생성이 포함된다.
- 사용자 기능 패키지는 `BFD-07-01-01-03`, `BFD-07-01-02-02`를 의존 기능으로 추적한다.
- 시스템관리 모듈은 인력 테이블을 직접 쓰지 않고 인력관리 애플리케이션 인터페이스를 통해서만 변경한다.
- 최초 등록의 선택 연락처 변경도 인력관리 개인정보 갱신 인터페이스를 호출하여 `TB_RES_MST`와 `TB_SYS_USER`가 불일치하지 않게 한다.
- 조직이 필수인 현재 직원 모델을 위해 부트스트랩에서 하나의 기본 조직을 생성한다. 이후 사용자는 등록된 조직 중 하나를 선택한다.
- 최상위 기본 조직의 유형은 초기 공통코드 `COMPANY`로 고정하고 조직명은 환경별 부트스트랩 입력값으로 받는다.
- 엑셀 입력을 추가할 때는 중복 사원번호, 행별 검증, 부분실패, 재처리와 개인정보 파일 보관·파기 정책을 별도로 설계해야 한다.
