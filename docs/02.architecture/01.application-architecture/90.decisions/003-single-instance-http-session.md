# ADR-003 단일 인스턴스 HttpSession 적용

## 상태

승인

## 일자

2026-07-22

## 배경

BMS v1.0은 약 50명 이내의 자사직원과 향후 로그인 가능한 외주인력이 사용하는 소규모 내부 시스템을 목표로 한다. 예상 확장 기능은 개인 학력·경력정보 입력 수준이며 초기 운영은 단일 백엔드 인스턴스로 충분하다.

공유 세션 저장소를 도입하면 다중 인스턴스와 무중단 배포에 유리하지만, 현재 규모에서는 별도 인프라 구축·운영·장애대응 비용이 기능 가치보다 크다.

## 결정

- v1.0 개발·운영환경은 서블릿 컨테이너 기본 `HttpSession`을 사용한다.
- 백엔드는 단일 인스턴스로 운영하고 서버 재시작·재배포 시 기존 세션이 소멸되는 것을 허용한다.
- 일반 세션 유휴시간은 세션에 별도로 저장한 `lastUserActivityAt` 기준 15분이다. 로그인 성공 시 초기화하고 유효한 활동 API만 갱신하며 다른 요청에 의해 바뀌는 컨테이너 접근시각은 인증 유휴시간 판정에 사용하지 않는다.
- 일반 세션은 사용자 활동과 관계없이 로그인 또는 제한 세션 승격 후 8시간에 절대 만료한다.
- 세션에는 사용자ID, 로그인ID, 표시명, 역할ID·역할명, 메뉴ID·상위메뉴ID·메뉴명·메뉴URL·표시순서, 접속로그ID, 비밀번호변경필요여부, `securityVersion`, `lastUserActivityAt`, `absoluteExpiresAt`과 `reauthenticatedAt`으로 구성된 인증·표시 스냅샷을 불변 DTO로 저장한다.
- JPA 엔터티, 비밀번호·해시, 세션ID·쿠키값, CSRF 토큰과 불필요한 개인정보는 세션 속성으로 저장하지 않는다.
- 세션 생성·조회·종료는 `AuthSessionService` 경계에 모으고 업무 코드가 세션 저장 구현에 직접 의존하지 않게 한다.
- 계정 비활성화 시 사용자별 세션을 찾아 즉시 종료하지 않는다. 모든 보호 API 요청에서 현재 계정의 활성·삭제 상태를 확인하여 다음 요청을 차단하고, 요청이 없으면 마지막 사용자 활동부터 최대 15분 안에 만료한다.
- 사용자 역할, 역할 메뉴권한, 역할 삭제 또는 비밀번호가 변경되면 대상 사용자의 `SEC_VER`를 증가시킨다. 모든 보호 API는 현재 값과 세션의 `securityVersion`을 비교하고 불일치 시 다음 요청을 차단한다.
- 초기 비밀번호 사용자의 제한 세션은 별도 `lastUserActivityAt` 기준 10분 유휴시간과 로그인 성공부터 30분의 절대 유효시간을 적용하며, 초기 비밀번호 만료일시가 더 빠르면 그 시각에 만료한다.
- 제한 세션은 최초 등록에 필요한 서버 측 API 허용목록만 사용할 수 있다. 활동 API는 유휴시간만 갱신하고 절대 유효시간은 연장하지 않는다.
- 최초 등록 완료 시 세션 식별자를 교체하고 일반 권한 스냅샷으로 대체하여 같은 로그인 흐름 안에서 일반 세션으로 승격한다.
- 로그인 요청 속도 제한은 `LoginAttemptLimiter` 인터페이스의 단일 인스턴스 메모리 구현을 사용한다.
- 중요 관리자 작업은 로그인 또는 현재 비밀번호 재인증 후 10분 안에서만 허용하며 재인증 요청은 세션별 분당 5회로 제한한다.
- 공유 세션 저장소는 현재 구축 범위에서 제외한다.

## 영향

- Redis 등 별도 세션 인프라 없이 로그인 기능을 구현할 수 있다.
- 서버 장애나 재배포 시 모든 사용자가 다시 로그인해야 한다.
- 애플리케이션 재기동 시 이전 실행의 미종료 성공 접속로그를 기동시각 기준 `TIMEOUT`으로 보정한다.
- 단일 인스턴스이므로 로드밸런서 분산과 인스턴스 간 세션 공유를 지원하지 않는다.
- 세션 전용 DTO와 서비스 경계를 지키면 향후 공유 저장소 도입 시 API와 화면 계약의 변경 없이 인프라 구성을 교체할 수 있다.

## 재검토 조건

다음 중 하나가 필요해지면 공유 세션 저장소 도입을 별도 ADR로 검토한다.

- 백엔드 2대 이상 운영
- 무중단 또는 순차 배포
- 서버 재시작 후 로그인 세션 유지
- 여러 인스턴스에서 로그인 속도 제한 카운터 공유
- 사용자별 전체 세션 조회와 관리자 강제 종료

사용자 수 증가나 개인 학력·경력 입력 기능 추가만으로는 재검토 조건이 충족된 것으로 보지 않는다.
