# BMS 데이터 카탈로그 스키마

## 1. 목적

본 문서는 데이터 카탈로그 CSV의 파일명, 헤더와 관리 규칙을 정의한다. CSV 헤더와 열 순서는 임의로 변경하지 않는다.

## 2. 공통 작성 규칙

- 모든 CSV는 UTF-8과 LF 줄바꿈으로 저장한다.
- 첫 번째 행은 고정된 헤더로 사용한다.
- 데이터 한 건은 한 행으로 관리한다.
- 셀 안에는 줄바꿈을 사용하지 않는다.
- 쉼표, 큰따옴표가 포함된 값은 CSV 인용 규칙을 적용한다.
- 업무영역별로 분리되는 CSV는 데이터가 없는 업무영역 파일도 헤더를 유지한다.
- 기존 `source_no`는 재번호하지 않는다.
- Excel은 CSV에서 생성하며 Excel 파일을 원본으로 수정하지 않는다.

## 3. 표준단어

파일은 `01.standard/standard-words.csv`이다.

```text
source_no,korean_name,english_name,abbreviation,category,description,note
```

## 4. 표준용어

파일은 `01.standard/standard-terms.csv`이다.

```text
source_no,owner_area,korean_name,english_name,column_name,domain_name,description,note
```

- `owner_area`는 용어를 사용하는 모든 영역이 아니라 용어의 의미와 변경을 책임지는 소유 업무영역이다.
- `owner_area`는 `system`, `common`, `customer`, `sales`, `contract`, `project`, `employee`, `cost`, `revenue` 중 하나를 사용한다.
- 여러 업무영역에서 사용하는 용어도 소유 업무영역의 용어 한 건만 등록한다.
- 같은 `column_name`을 사용하더라도 한글명과 업무 의미가 다르면 별도 용어로 등록할 수 있다.

## 5. 논리 도메인

파일은 `01.standard/domain-definition.csv`이다.

```text
record_type,category,domain_name,allowed_value,nullable,default_value,description,example_column
```

- `record_type=DOMAIN`은 논리 도메인을 의미한다.
- `record_type=ALLOWED_VALUE`는 도메인의 허용값을 의미한다.
- DBMS 물리 타입은 이 파일에 기록하지 않는다.

## 6. DBMS 타입 매핑

파일은 `01.standard/db-type-mapping.csv`이다.

```text
dbms,domain_name,data_type
```

## 7. 엔터티

파일은 `02.logical-model/entities/entity-{business-area}.csv` 형식을 사용한다.

```text
entity_name,entity_type,description,note
```

## 8. 엔터티 속성

파일은 `02.logical-model/attributes/entity-attribute-{business-area}.csv` 형식을 사용한다.

```text
entity_name,attribute_sequence,attribute_name
```

`attribute_sequence`는 엔터티 안에서 기존 주요 속성 후보의 기재 순서를 보존한다.

### 8.1 외부 연계 수신 속성

연계기관이 정의한 응답 필드명을 원형 그대로 보존해야 하는 엔터티는 `02.logical-model/attributes/interface-attribute-{business-area}.csv` 형식으로 관리한다.

```text
entity_name,attribute_sequence,attribute_name,source_attribute_name,standard_term_exception_yn,description,note
```

- BMS 내부 처리 속성은 `source_attribute_name`을 비우고 `standard_term_exception_yn=N`으로 관리하며 등록된 표준용어를 사용한다.
- 연계기관 응답 속성은 `source_attribute_name`에 원본 필드명을 기록하고 `standard_term_exception_yn=Y`로 관리한다.
- `standard_term_exception_yn=Y`인 속성은 표준단어·표준용어 등록 및 물리 컬럼 명명규칙 검사의 예외로 하며, 연계 명세의 영문명과 국문명을 변경하지 않는다.
- 외부 연계 수신 속성의 유일성은 `entity_name + source_attribute_name`으로 검사한다. 연계 명세에 동일한 국문명이 있어도 서로 다른 원본 필드명이면 허용한다.
- 예외는 외부 응답 필드에만 적용하며 식별자, 처리상태, 감사 속성 등 BMS 내부 속성에는 적용하지 않는다.

## 9. 엔터티 관계

업무영역 내부 관계는 `02.logical-model/relationships/entity-relation-{business-area}.csv`, 영역 간 관계는 `02.logical-model/relationships/entity-relation-cross.csv`에서 관리한다.

```text
relation_expression,description
```

- 현재 관계 CSV는 논리 모델의 관계 후보와 역할을 설명하는 문서이다.
- 검증기는 파일 구조와 관계식 형식만 검사하며 관계식의 명칭을 엔터티 참조로 검사하지 않는다.
- 테이블, PK와 FK 컬럼이 확정된 뒤 물리 모델에서 참조 무결성을 검사한다.

## 10. 물리 테이블

물리 설계를 시작한 업무영역은 `03.physical-model/tables/table-{business-area}.csv` 형식을 사용한다.

```text
entity_name,table_name,table_type,description,note
```

- `entity_name`은 논리 엔터티명이다.
- `table_name`은 DB 명명규칙을 적용한 물리 테이블명이다.
- `table_type`은 `MASTER`, `BUSINESS`, `RELATION`, `HISTORY`, `LOG` 중 하나를 사용한다.

## 11. 물리 컬럼

물리 설계를 시작한 업무영역은 `03.physical-model/columns/column-{business-area}.csv` 형식을 사용한다.

```text
table_name,column_sequence,korean_name,column_name,domain_name,data_type_override,nullable,default_value,pk_yn,description,note
```

- `korean_name`은 등록된 표준용어의 한글명이다.
- `domain_name`은 논리 도메인을 참조한다.
- 물리 데이터 타입은 `db-type-mapping.csv`에서 파생하고 DBMS 예외가 있을 때만 `data_type_override`를 사용한다.
- `nullable`, `pk_yn`은 `Y` 또는 `N`을 사용한다.
- `column_sequence`는 테이블 안의 물리 컬럼 기재 순서이다.

## 12. 물리 제약조건과 참조

물리 설계를 시작한 업무영역은 `03.physical-model/constraints/constraint-{business-area}.csv` 형식을 사용한다.

```text
constraint_name,constraint_type,table_name,column_names,reference_table,reference_columns,check_expression,on_delete,on_update,enforcement_type,create_yn,description
```

- `constraint_type`은 `PK`, `UK`, `FK`, `CK` 중 하나를 사용한다.
- 복합 컬럼의 `column_names`와 `reference_columns`는 기재 순서대로 `|`로 구분한다.
- `enforcement_type`은 DB 제약조건으로 집행하면 `DATABASE`, 애플리케이션에서 집행하면 `APPLICATION`을 사용한다.
- `create_yn=Y`는 DDL에 제약조건을 생성하고 `create_yn=N`은 참조 메타데이터만 관리한다.
- `DATABASE`는 `create_yn=Y`, `APPLICATION`은 `create_yn=N`을 사용한다.
- `reference_table`, `reference_columns`, `on_delete`, `on_update`는 FK에 사용한다.
- `check_expression`은 CK에 사용한다.

## 13. 물리 인덱스

물리 설계를 시작한 업무영역은 `03.physical-model/indexes/index-{business-area}.csv` 형식을 사용한다.

```text
index_name,table_name,column_names,unique_yn,index_method,where_condition,description
```

- 복합 인덱스의 `column_names`는 기재 순서대로 `|`로 구분한다.
- `unique_yn`은 `Y` 또는 `N`을 사용한다.
- `index_method`가 비어 있으면 PostgreSQL `BTREE`를 기본값으로 사용한다.
- 부분 인덱스가 필요한 경우에만 `where_condition`을 사용한다.
