# BMS DB 명명규칙

## 1. 문서 개요

### 1.1 목적

본 문서는 **BMS(Business Management System)** 데이터베이스 설계 시 적용할 명명규칙(Naming Rule)을 정의한다.

명명규칙은 데이터 모델의 일관성을 확보하고 개발 및 유지보수의 효율성을 높이기 위한 기준으로 활용한다.

본 문서는 다음 DB 객체에 대한 명명규칙을 정의한다.

* 테이블(Table)
* 컬럼(Column)
* Primary Key
* Foreign Key
* Index
* Sequence
* View
* Trigger

---

## 2. 기본 원칙

### 2.1 공통 원칙

* 모든 DB 객체명은 **영문 대문자**를 사용한다.
* 단어 구분은 **언더스코어(_)를 사용**한다.
* 공백 및 특수문자는 사용하지 않는다.
* 약어는 표준 약어를 사용한다.
* 동일한 의미의 데이터는 동일한 명칭을 사용한다.
* Oracle, MariaDB 등 DBMS에 관계없이 동일한 규칙을 적용한다.

예)

- 구분: 테이블; 사용: `TB_PRJ_MST`
- 구분: 컬럼; 사용: `PRJ_ID`
- 구분: PK; 사용: `PK_TB_PRJ_MST`
- 구분: FK; 사용: `FK_TB_PRJ_MEMBER_01`

# 3. 업무영역 Prefix

업무영역을 구분하기 위해 다음 Prefix를 사용한다.

- 업무영역: 시스템관리; Prefix: SYS
- 업무영역: 공통; Prefix: COM
- 업무영역: 고객관리; Prefix: CUST
- 업무영역: 영업관리; Prefix: SALES
- 업무영역: 계약관리; Prefix: CNTR
- 업무영역: 프로젝트관리; Prefix: PRJ
- 업무영역: SI 프로젝트; Prefix: SIPRJ
- 업무영역: SM 프로젝트; Prefix: SMPRJ
- 업무영역: 인력관리; Prefix: RES
- 업무영역: 예산·원가관리; Prefix: COST
- 업무영역: 매출·수금관리; Prefix: REV

# 4. DB 객체 명명규칙

## 4.1 테이블(Table)

### 기본 형식

```text
TB_업무영역_엔터티[_유형]
```

### 예시

- 테이블명: TB_SYS_USER; 설명: 사용자
- 테이블명: TB_SYS_MENU; 설명: 메뉴
- 테이블명: TB_COM_CODE; 설명: 공통코드
- 테이블명: TB_CUST_MST; 설명: 고객사
- 테이블명: TB_CUST_CONTACT; 설명: 고객 담당자
- 테이블명: TB_SALES_OPPORTUNITY; 설명: 영업기회
- 테이블명: TB_CNTR_MST; 설명: 계약
- 테이블명: TB_CNTR_CHANGE_HIST; 설명: 계약변경이력
- 테이블명: TB_PRJ_MST; 설명: 프로젝트
- 테이블명: TB_PRJ_MEMBER; 설명: 프로젝트 참여인력
- 테이블명: TB_PRJ_SCHEDULE; 설명: 프로젝트 일정
- 테이블명: TB_PRJ_ISSUE; 설명: 프로젝트 이슈
- 테이블명: TB_PRJ_RISK; 설명: 프로젝트 위험
- 테이블명: TB_PRJ_OUTPUT; 설명: 프로젝트 산출물
- 테이블명: TB_RES_EMPLOYEE; 설명: 직원
- 테이블명: TB_RES_ASSIGN; 설명: 프로젝트 투입
- 테이블명: TB_COST_BUDGET; 설명: 예산
- 테이블명: TB_COST_EXPENSE; 설명: 경비
- 테이블명: TB_REV_BILL; 설명: 청구
- 테이블명: TB_REV_RECEIPT; 설명: 수금

> **업무영역 Prefix와 동일한 의미의 단어는 중복 사용하지 않는다.**
>
> 예)
>
> * ○ `TB_PRJ_MST`
> * × `TB_PRJ_PROJECT_MST`

# 6. 공통 컬럼

모든 업무 테이블은 다음 공통 컬럼을 사용한다.

- 컬럼명: REG_ID; 설명: 등록자
- 컬럼명: REG_DTM; 설명: 등록일시
- 컬럼명: MOD_ID; 설명: 수정자
- 컬럼명: MOD_DTM; 설명: 수정일시
- 컬럼명: DEL_YN; 설명: 삭제여부

로그 및 불변 이력 테이블은 REG_ID, REG_DTM만 적용할 수 있다.
순수 관계 테이블은 업무 필요에 따라 MOD 계열 및 DEL_YN 적용을 생략할 수 있다.
위와 같은 이유로 다음 테이블에는 예외 적용한다.

 - 로그인로그
 - 감사로그
 - 비밀번호 이력
 - 사용자역할 관계
 - 역할메뉴권한 관계

### 삭제 정책

* 물리 삭제를 지양한다.
* 논리 삭제를 기본으로 한다.
* 삭제 여부는 `DEL_YN` 컬럼을 사용한다.
* 값은 `Y`, `N`만 사용한다.
* 동일 테이블에 `USE_YN`과 `DEL_YN`을 함께 두지 않는다.
* 상태코드는 업무 생명주기를, `USE_YN`은 삭제여부가 없는 단순 기준정보의 선택 가능 여부를 나타낸다.
* 상세 기준은 `03.physical-model-common-policy.md`를 따른다.

# 7. Primary Key

PK 컬럼은 엔터티 약어 + ID를 사용한다.

내부 엔터티 ID는 애플리케이션에서 생성한 26자리 Monotonic ULID를 사용한다. 외부 식별자와 업무번호는 별도 컬럼으로 관리한다.

- 테이블: TB_PRJ_MST; PK: PRJ_ID
- 테이블: TB_CUST_MST; PK: CUST_ID
- 테이블: TB_CNTR_MST; PK: CNTR_ID
- 테이블: TB_RES_EMPLOYEE; PK: EMP_ID

### PK 제약조건

```text
PK_테이블명
```

예)

```text
PK_TB_PRJ_MST
PK_TB_CUST_MST
```

---

# 8. Foreign Key

FK 컬럼은 참조 테이블의 PK 컬럼명을 그대로 사용한다.

예)

- 참조 테이블: 고객; FK 컬럼: CUST_ID
- 참조 테이블: 프로젝트; FK 컬럼: PRJ_ID
- 참조 테이블: 계약; FK 컬럼: CNTR_ID
- 참조 테이블: 직원; FK 컬럼: EMP_ID

### FK 제약조건

```text
FK_테이블명_순번
```

예)

```text
FK_TB_PRJ_MEMBER_01
FK_TB_PRJ_MEMBER_02
```

---

# 9. Index

### 일반 Index

```text
IX_테이블명_순번
```

예)

```text
IX_TB_PRJ_MST_01
```

### Unique Index

```text
UX_테이블명_순번
```

예)

```text
UX_TB_SYS_USER_01
```

논리삭제 테이블의 고유 인덱스는 업무키 재사용 가능 여부에 따라 전체 UNIQUE 또는 `DEL_YN = 'N'` 부분 UNIQUE로 구분하며 상세 기준은 `03.physical-model-common-policy.md`를 따른다.

# 11. 테이블 유형

필요한 경우 다음 Suffix를 사용할 수 있다.

- Suffix: MST; 설명: Master
- Suffix: DTL; 설명: Detail
- Suffix: HIST; 설명: History
- Suffix: REL; 설명: Relation
- Suffix: LOG; 설명: Log
- Suffix: FILE; 설명: 첨부파일
- Suffix: STAT; 설명: 통계
- Suffix: TMP; 설명: 임시

예)

```text
TB_CNTR_CHANGE_HIST
TB_PRJ_MEMBER_REL
TB_SYS_LOGIN_LOG
```
