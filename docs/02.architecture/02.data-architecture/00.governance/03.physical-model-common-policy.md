# 전체 물리모델 공통 기준

## 1. 목적

본 문서는 PostgreSQL 물리모델에 공통 적용하는 ID 생성, 시간 저장, 참조 무결성, 코드 초기 데이터, 상태와 논리삭제 및 고유 인덱스 기준을 정의한다.

## 2. ID 생성

- 내부 엔터티의 단일 식별자는 애플리케이션 공통 생성기가 INSERT 전에 생성한 Monotonic ULID를 사용한다.
- ULID는 26자리 영문 대문자와 숫자의 정규형으로 저장하며 PostgreSQL 타입은 `VARCHAR(26)`을 사용한다.
- DB 시퀀스, 트리거와 컬럼 DEFAULT로 내부 엔터티 ID를 생성하지 않는다.
- FK에는 부모 PK의 ULID 값을 그대로 사용한다.
- 외부기관 식별자와 업무번호는 ULID로 변환하지 않고 별도 외부 식별자 또는 번호 속성에 원문을 저장한다.
- 로그인ID와 같이 이름에 ID가 포함되지만 업무 의미가 있는 문자열은 ULID 생성 대상이 아니며 해당 업무 검증 규칙을 적용한다.
- 등록자ID와 수정자ID에는 사용자 ULID 또는 `SYSTEM`과 같은 예약 처리주체 값을 저장할 수 있다.

## 3. 시간 저장

- 일시 도메인의 PostgreSQL 타입은 `TIMESTAMP`를 사용한다.
- DB에 저장하는 일시는 UTC를 기준으로 한다.
- DB 서버, 연결 세션, 애플리케이션 기본 시간대를 UTC로 고정하고 `CURRENT_TIMESTAMP`가 UTC 기준으로 저장되도록 한다.
- 외부 연계 일시는 원본 시간대를 확인한 뒤 UTC로 변환하여 저장한다.
- 화면과 보고서는 UTC 값을 `Asia/Seoul`로 변환하여 표시한다.
- API 일시는 시간대 오프셋을 포함한 ISO 8601 형식으로 송수신한다.
- `DATE`, `YEAR`, `YEARMONTH`는 업무 달력값이므로 시간대 변환을 적용하지 않는다.

## 4. 참조 무결성

### 4.1 기본 원칙

- 논리 FK는 모두 물리 제약조건 CSV에 기록한다.
- 기본 집행 방식은 `enforcement_type=APPLICATION`, `create_yn=N`이다.
- DB FK를 생성하는 관계는 `enforcement_type=DATABASE`, `create_yn=Y`로 기록한다.
- DB FK의 삭제·수정 동작은 모두 `RESTRICT`로 하고 자동 `CASCADE`는 사용하지 않는다.
- 논리삭제된 부모의 참조 가능 여부는 DB FK가 판별할 수 없으므로 애플리케이션에서 `DEL_YN='N'`을 검증한다.

### 4.2 DB FK 필수 대상

다음 관계는 부모가 없으면 자식의 의미가 성립하지 않거나 인증·권한·정산 무결성에 직접 영향을 주므로 DB FK를 생성한다.

- 공유 PK 하위유형: 직원·외주인력에서 인력
- 인증·권한 관계: 사용자역할에서 사용자·역할, 역할메뉴권한에서 역할·메뉴
- 코드 종속관계: 공통코드에서 코드그룹
- 공통 업무대상 관계: 첨부파일·메모에서 업무대상, 업무대상 적용 엔터티에서 업무대상
- 인력 종속관계: 인력이력·인력기술경력·인력단가에서 인력, 인력투입에서 프로젝트·인력, 인력월별투입실적에서 인력투입
- 영업 핵심 진행관계: 사업성분석에서 영업기회, 제안에서 사업공고, 수주에서 제안, 재공고에서 원공고
- 계약·정산 필수 상세관계: 계약 상세에서 계약, 예산상세에서 예산, 청구·수금·세금계산서 등 독립 존재할 수 없는 거래 상세에서 직전 부모

### 4.3 DB FK 제외 대상

- 등록자ID·수정자ID·처리사용자ID 등 감사 속성
- 접속로그·시스템로그·API호출로그의 사용자 등 로그 보존기간과 독립적으로 관리해야 하는 참조
- 시스템로그의 `(대상유형코드, 대상ID)` 일반 감사대상 다형 참조
- 외부기관 식별자
- 부모의 보존기간과 독립적으로 보관해야 하는 로그 참조
- 조회 편의를 위한 선택적·파생 참조

### 4.4 공통 업무대상

- 첨부파일과 메모는 `TASK_TGT_ID`로 `TB_COM_TASK_TGT`를 필수 참조하며 `(TASK_SE_CD, TASK_ID)` 다형 참조를 사용하지 않는다.
- 하나의 첨부파일은 하나의 업무대상에만 소속하며 여러 업무대상과 공유하기 위한 별도 관계 테이블을 생성하지 않는다.
- 업무대상 적용 엔터티는 `TASK_TGT_ID`로 `TB_COM_TASK_TGT`를 필수 참조하고 전체 UNIQUE를 적용하여 하나의 업무대상에 같은 엔터티 행이 중복 연결되지 않도록 한다.
- 업무대상유형코드와 실제 연결 엔터티의 일치 및 하나의 업무대상이 서로 다른 유형의 엔터티에 중복 연결되지 않는지는 애플리케이션 트랜잭션에서 검증한다.
- 업무대상 레코드는 등록·수정·논리삭제 공통 컬럼을 사용하는 일반 업무 엔터티로 관리한다. 첨부파일·메모 또는 적용 대상 엔터티가 참조 중이면 물리삭제하지 않는다.
- 적용 대상 엔터티를 논리삭제해도 업무대상과 기존 첨부파일·메모는 보존하며, 논리삭제된 업무대상에 대한 신규 첨부파일·메모 등록은 애플리케이션에서 차단한다.
- 계약문서는 계약과 별개의 업무대상으로 등록한다. 계약문서가 참조하는 PDF 첨부파일의 업무대상ID는 계약문서의 업무대상ID와 일치해야 하며 애플리케이션 트랜잭션에서 검증한다.
- 시스템로그의 일반 감사대상은 사용자·역할 등 공통 업무대상에 포함되지 않는 대상도 기록하므로 `TB_COM_TASK_TGT` 참조로 전환하지 않는다.

## 5. 코드값과 초기 데이터

- 공통 코드값은 코드그룹과 공통코드에서 관리한다.
- 코드값은 영문 대문자 스네이크 형식을 사용하며 운영에서 사용된 뒤에는 변경하거나 재사용하지 않는다.
- 코드명·설명·정렬순서는 변경할 수 있으나 사용 이력이 있는 코드값은 물리 삭제하지 않는다.
- 삭제된 코드는 신규 입력 선택지에서 제외하고 기존 데이터 조회 시 명칭을 확인할 수 있어야 한다.
- 필수 초기 코드와 시스템 설정은 버전 관리되는 DB 마이그레이션으로 등록한다.
- 개발·테스트·운영 환경은 동일한 코드값을 사용한다.
- 운영에서 긴급 등록한 코드도 다음 배포 전에 마이그레이션 원본에 반영한다.
- `Y/N`과 변경 가능성이 없는 소규모 내부 처리값은 공통코드 대신 CHECK 제약조건을 사용할 수 있다.

## 6. 상태·사용여부·삭제여부

- 상태코드는 업무 생명주기의 현재 단계를 나타내며 유지한다.
- 사용여부는 상태코드와 삭제여부가 모두 없는 단순 기준정보의 선택 가능 여부에만 사용한다.
- 동일 엔터티에 사용여부와 삭제여부가 함께 있으면 사용여부를 제거하고 삭제여부를 단일 기준으로 사용한다.
- 상태코드가 있는 엔터티의 비활성·종료·취소는 상태코드로 표현한다.
- 로그와 불변 이력처럼 삭제하지 않는 엔터티에는 삭제여부를 두지 않는다.
- 삭제여부는 `NOT NULL DEFAULT 'N'`과 `DEL_YN IN ('Y','N')` CHECK를 적용한다.

## 7. 논리삭제와 고유 인덱스

- 사용자 로그인ID, 사원번호, 사업자등록번호, 계약번호와 외부 공고 식별번호처럼 재사용하면 안 되는 업무 식별자는 삭제여부와 관계없이 전체 UNIQUE를 적용한다.
- 역할명, 관리용 명칭, 기간별 이력 조합과 재생성 가능한 관계키는 활성 행만 고유하도록 `WHERE DEL_YN = 'N'` 부분 UNIQUE 인덱스를 적용한다.
- 부분 인덱스 조건은 특별한 사유가 없으면 `DEL_YN = 'N'`으로 통일한다.
- 논리삭제된 행을 복구할 때도 동일한 고유성 검사를 적용한다.
- 개별 컬럼의 `unique_yn`으로 표현할 수 없는 복합 고유성은 물리 인덱스 CSV에 구성 컬럼과 조건을 명시한다.
