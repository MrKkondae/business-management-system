# 공통 업무영역 상세 ERD

## 1. 문서 개요

여러 업무영역에서 재사용하는 알림, 업무대상, 첨부파일 및 메모의 PostgreSQL 물리 모델을 표현한다. 원본은 데이터 카탈로그 CSV이며 이 문서는 구현과 리뷰를 위한 파생 산출물이다.

- 기준 DBMS: PostgreSQL
- 범위: 공통 업무영역 4개 테이블
- 표기: `PK`는 기본키, `FK`는 논리 참조 컬럼, `DB FK`는 DB 제약 집행, `APP REF`는 애플리케이션 집행, `LOGICAL REF`는 상대 영역 물리화 전 논리 관계
- 타입 표기: Mermaid 호환을 위해 `VARCHAR(26)`은 `VARCHAR_26`, `CHAR(1)`은 `CHAR_1`처럼 괄호를 밑줄로 표시
- 카디널리티: `||` 필수 1, `o|` 선택 1, `o{` 0개 이상

### 1.1 원본 카탈로그

- 테이블: `03.physical-model/tables/table-common.csv`
- 컬럼: `03.physical-model/columns/column-common.csv`
- 제약조건: `03.physical-model/constraints/constraint-common.csv`
- 인덱스: `03.physical-model/indexes/index-common.csv`
- 타입 매핑: `01.standard/db-type-mapping.csv`

### 1.2 업무기능 추적성

| 기능 ID | 업무기능 | 주요 테이블 |
| --- | --- | --- |
| BFD-02-01 | 대시보드 | 업무영역별 원천 데이터의 파생 조회 |
| BFD-02-02 | 알림관리 | TB_COM_NOTI |
| BFD-02-03 | 첨부파일관리 | TB_COM_TASK_TGT, TB_COM_FILE |
| BFD-02-04 | 메모관리 | TB_COM_TASK_TGT, TB_COM_MEMO |
| BFD-02-05 | 인증관리 | TB_SYS_USER, TB_SYS_ACCESS_LOG |

## 2. 전체 관계 개요

```mermaid
erDiagram
    TB_SYS_USER ||--o{ TB_COM_NOTI : "APP REF: RCVR_ID"
    TB_COM_TASK_TGT ||--o{ TB_COM_FILE : "DB FK: TASK_TGT_ID"
    TB_COM_TASK_TGT ||--o{ TB_COM_MEMO : "DB FK: TASK_TGT_ID"
    TB_SYS_USER ||--o{ TB_COM_MEMO : "APP REF: WRTR_ID"
    TB_COM_TASK_TGT ||--o| CONTRACT_LOGICAL : "LOGICAL REF: TASK_TGT_ID"
    TB_COM_TASK_TGT ||--o| TB_PRJ_MST : "DB FK: TASK_TGT_ID"
    TB_COM_TASK_TGT ||--o| TB_SALES_OPPTY : "DB FK: TASK_TGT_ID"
    TB_COM_TASK_TGT ||--o| CONTRACT_DOCUMENT_LOGICAL : "LOGICAL REF: TASK_TGT_ID"
    TB_COM_TASK_TGT ||--o| TB_SALES_ANNC : "DB FK: TASK_TGT_ID"
    TB_COM_TASK_TGT ||--o| TB_SALES_PRPSL : "DB FK: TASK_TGT_ID"
    TB_COM_TASK_TGT ||--o| TB_RES_MST : "DB FK: TASK_TGT_ID"
    TB_COM_FILE ||--o| CONTRACT_DOCUMENT_LOGICAL : "LOGICAL REF: PDF_FILE_ID"
```

> 계약·계약문서는 아직 물리 모델이 확정되지 않아 `*_LOGICAL`로 표시했다. 프로젝트·영업기회·사업공고·제안·인력의 물리 테이블은 `TASK_TGT_ID`를 필수·고유 참조한다. 업무대상 유형과 실제 연결 엔터티의 일치는 애플리케이션 트랜잭션에서 검증한다.

## 3. 영역별 상세 ERD

### 3.1 알림

사용자별 업무 이벤트 알림과 읽음 상태를 관리하는 구조이다.

```mermaid
erDiagram
    TB_COM_NOTI {
        VARCHAR_26 NOTI_ID PK "알림ID; NOT NULL"
        VARCHAR_26 RCVR_ID FK "수신자ID; NOT NULL"
        VARCHAR_20 NOTI_TYPE_CD "알림유형; NOT NULL"
        VARCHAR_200 TITLE "제목; NOT NULL"
        TEXT CONT "내용; NULL"
        CHAR_1 READ_YN "읽음여부; NOT NULL, DEFAULT 'N'"
        TIMESTAMP OCCUR_DTM "발생일시; NOT NULL, DEFAULT CURRENT_TIMESTAMP"
        VARCHAR_26 REG_ID "등록자ID; NOT NULL"
        TIMESTAMP REG_DTM "등록일시; NOT NULL, DEFAULT CURRENT_TIMESTAMP"
        VARCHAR_26 MOD_ID "수정자ID; NULL"
        TIMESTAMP MOD_DTM "수정일시; NULL"
        CHAR_1 DEL_YN "삭제여부; NOT NULL, DEFAULT 'N'"
    }
    TB_SYS_USER {
        ID USER_ID PK "외부 참조 식별자"
    }
    TB_SYS_USER ||--o{ TB_COM_NOTI : "APP REF: RCVR_ID"
```

테이블 대응:
- `TB_COM_NOTI`: 알림

### 3.2 업무대상·첨부파일·메모

공통 업무대상을 중심으로 파일과 메모를 연결하는 범용 구조이다.

```mermaid
erDiagram
    TB_COM_TASK_TGT {
        VARCHAR_26 TASK_TGT_ID PK "업무대상ID; NOT NULL"
        VARCHAR_20 TASK_TGT_TYPE_CD "업무대상유형코드; NOT NULL"
        VARCHAR_26 REG_ID "등록자ID; NOT NULL"
        TIMESTAMP REG_DTM "등록일시; NOT NULL, DEFAULT CURRENT_TIMESTAMP"
        VARCHAR_26 MOD_ID "수정자ID; NULL"
        TIMESTAMP MOD_DTM "수정일시; NULL"
        CHAR_1 DEL_YN "삭제여부; NOT NULL, DEFAULT 'N'"
    }
    TB_COM_FILE {
        VARCHAR_26 FILE_ID PK "파일ID; NOT NULL"
        VARCHAR_26 TASK_TGT_ID FK "업무대상ID; NOT NULL"
        VARCHAR_200 ORGNL_FILE_NM "원본파일명; NOT NULL"
        VARCHAR_200 SAVE_FILE_NM "저장파일명; NOT NULL"
        VARCHAR_1000 FILE_PATH "파일경로; NOT NULL"
        BIGINT FILE_SIZE "파일크기; NOT NULL"
        VARCHAR_26 REG_ID "등록자ID; NOT NULL"
        TIMESTAMP REG_DTM "등록일시; NOT NULL, DEFAULT CURRENT_TIMESTAMP"
        VARCHAR_26 MOD_ID "수정자ID; NULL"
        TIMESTAMP MOD_DTM "수정일시; NULL"
        CHAR_1 DEL_YN "삭제여부; NOT NULL, DEFAULT 'N'"
    }
    TB_COM_MEMO {
        VARCHAR_26 MEMO_ID PK "메모ID; NOT NULL"
        VARCHAR_26 TASK_TGT_ID FK "업무대상ID; NOT NULL"
        TEXT CONT "내용; NOT NULL"
        VARCHAR_26 WRTR_ID FK "작성자ID; NOT NULL"
        TIMESTAMP WRT_DTM "작성일시; NOT NULL, DEFAULT CURRENT_TIMESTAMP"
        VARCHAR_26 REG_ID "등록자ID; NOT NULL"
        TIMESTAMP REG_DTM "등록일시; NOT NULL, DEFAULT CURRENT_TIMESTAMP"
        VARCHAR_26 MOD_ID "수정자ID; NULL"
        TIMESTAMP MOD_DTM "수정일시; NULL"
        CHAR_1 DEL_YN "삭제여부; NOT NULL, DEFAULT 'N'"
    }
    TB_SYS_USER {
        ID USER_ID PK "외부 참조 식별자"
    }
    TB_COM_TASK_TGT ||--o{ TB_COM_FILE : "DB FK: TASK_TGT_ID"
    TB_COM_TASK_TGT ||--o{ TB_COM_MEMO : "DB FK: TASK_TGT_ID"
    TB_SYS_USER ||--o{ TB_COM_MEMO : "APP REF: WRTR_ID"
```

테이블 대응:
- `TB_COM_TASK_TGT`: 업무대상
- `TB_COM_FILE`: 첨부파일
- `TB_COM_MEMO`: 메모

## 7. 구현 주의사항

- 업무대상은 허용된 구체 엔터티 하나와만 연결하고 유형코드와 실제 엔터티의 일치를 애플리케이션에서 검증한다.
- 논리삭제된 업무대상에는 새 첨부파일과 메모를 등록하지 않으며 기존 데이터는 보존 정책에 따라 조회한다.
- 첨부파일은 하나의 업무대상에만 소속하며 등록 후 업무대상ID를 변경하지 않는다.
- 파일경로와 저장파일명은 시스템이 생성하고 크기·확장자·실제 형식 및 정규화된 저장경로를 검증한다.
- 계약문서의 PDF 첨부파일과 계약문서는 동일한 업무대상ID를 사용해야 한다.
- 알림은 수신자 본인만 조회·읽음 처리하고 메모 수정·삭제는 작성자 또는 관리권한 보유자만 수행한다.
