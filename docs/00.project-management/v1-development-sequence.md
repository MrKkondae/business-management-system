# v1 시스템·인력·사용자관리 개발순서

## 1. 문서 목적

인증과 공통 애플리케이션 이후 시스템 기준정보, 인력등록과 사용자관리의
의존관계를 기준으로 v1 구현 순서를 정한다.

## 2. 현재 완료 기준

- 로그인, 로그아웃, 세션·CSRF와 인증 보안
- 최초 시스템관리자 부트스트랩
- 최초 등록과 초기 비밀번호 변경
- 공통 애플리케이션 기본 화면 `COM-006`
- 역할·메뉴·공통코드의 인증용 최소 초기 데이터

초기 데이터가 있으므로 아래 시스템관리 화면이 없어도 개발 DB에서 사용자등록을
기술적으로 시험할 수 있다. 그러나 운영자가 조직·역할·메뉴·코드를 관리할 수 있는
기준정보 기능을 사용자관리보다 먼저 구현하는 편이 운영 순서와 의존관계에 맞다.

## 3. 권장 개발 흐름

```text
인증·공통 애플리케이션 [완료]
          │
          ├─ A. SYS-004 공통코드관리
          │          └─ SYS-005 조직관리
          │
          └─ B. SYS-003 메뉴관리
                     └─ SYS-002 역할·메뉴권한관리

        A·B 기준정보 완료
                 │
                 ▼
인력등록 백엔드 완성 단위
  - TaskTargetRegistrationUseCase
  - EmployeeRegistrationUseCase
  - EmployeePersonalInfoUpdateUseCase
  - 인력·직원 저장소와 트랜잭션 테스트
                 │
                 ▼
SYS-001 사용자 목록·등록 백엔드
                 │
                 ▼
SYS-001 사용자 목록·등록 프런트엔드
                 │
                 ▼
RES-001 직원관리 전체 화면·후속 기능
```

`A`와 `B` 갈래는 서로 직접 의존하지 않으므로 설계와 구현을 병행할 수 있다.

## 4. 단계별 범위

### 1단계: SYS-004 공통코드관리

- 코드그룹·코드 조회, 등록, 수정과 논리삭제
- 계정상태, 조직유형, 인력구분과 재직상태 표시 기준
- 사용 중 코드의 삭제 제한과 감사로그

공통코드는 조직과 인력 상태값의 기준이므로 조직·인력 화면보다 먼저 구현한다.

### 2단계: SYS-003 메뉴관리

- 계층형 메뉴 조회·등록·수정·논리삭제
- URL과 표시순서 관리
- 구현되지 않은 URL을 활성 메뉴로 노출하지 않는 검증

역할별 메뉴권한은 메뉴 기준정보를 참조하므로 역할·권한관리보다 먼저 구현한다.

### 3단계: SYS-005 조직관리

- 조직 트리 조회, 등록, 수정과 논리삭제
- 상위조직·조직유형 검증
- 하위조직 또는 소속 직원이 있는 조직의 삭제 제한

사용자와 직원 등록은 미삭제 조직을 필수 참조하므로 `SYS-001`보다 먼저 구현한다.

### 4단계: SYS-002 역할·메뉴권한관리

- 역할 기본관리
- 역할별 메뉴권한 조회·저장
- 역할 사용자 관계 관리
- 변경된 역할·메뉴권한의 다음 로그인 반영

사용자 등록은 초기 역할을 한 건 이상 지정하므로 `SYS-001`보다 먼저 구현한다.

### 5단계: 인력등록 백엔드 완성 단위

이 단계는 독립 `RES-001` 화면 전체가 아니라 사용자등록이 호출할 인력 모듈의
공개 애플리케이션 인터페이스를 먼저 구현하는 단계다.

- 공통 업무대상 생성
- 인력 마스터와 직원의 공유 PK 생성
- 사원번호 전체 고유성 검증
- 신규 직원 최소정보 등록
- 개인정보 기준 원본 갱신
- 호출자 트랜잭션 참여와 전체 롤백 테스트

시스템 모듈은 인력 테이블에 직접 접근하지 않는다.

### 6단계: SYS-001 사용자 목록·등록

- 사용자 목록·필터·페이징과 개인정보 마스킹
- 조직·역할 등록 옵션
- 직원 등록 유스케이스 호출 후 사용자·초기 역할 생성
- 임시 비밀번호 1회 발급
- 최초 등록 연락처의 인력·사용자 동기화 보완
- 개발 DB 통합 검증

### 7단계: RES-001 직원관리 전체 기능

- 직원·인력 목록과 상세
- 입사일자·직위·재직상태 등 상세정보 수정
- 사용자 계정 연결상태 조회
- 기존 직원에 사용자 계정을 추가로 연결하는 후속 정책 검토

## 5. 인력등록 선행 범위의 해석

인력등록이 사용자관리보다 먼저라는 원칙은 맞다. 다만 두 가지를 구분한다.

1. **인력등록 백엔드 유스케이스 선행**  
   현재 확정안과 일치한다. 사용자등록 화면이 신규 직원 최소정보를 받아 인력 모듈을
   호출한다.
2. **독립 RES-001 직원등록 화면 선행**  
   직원이 사용자보다 먼저 존재하게 되므로 현재 확정한 “SYS-001은 신규 직원을
   항상 함께 생성하고 기존 사원번호를 거부”하는 정책과 충돌한다. 이 순서를
   선택하려면 `SYS-001`에 기존 직원 검색·선택·계정 연결 기능을 추가하고
   `D-011`을 다시 결정해야 한다.

v1 권장안은 1번이다. 인력 모듈의 등록·개인정보 갱신 인터페이스와 테스트를 먼저
완성하되, `RES-001` 전체 화면은 `SYS-001` 이후에 구현한다.

## 6. 단계 진입 조건

| 다음 단계 | 선행 완료 조건 |
| --- | --- |
| `SYS-005` | 조직유형 공통코드를 관리할 수 있음 |
| `SYS-002` | 메뉴 기준정보와 계층·URL 검증 완료 |
| 인력등록 백엔드 | 조직 조회 계약과 인력 관련 공통코드 확정 |
| `SYS-001` 백엔드 | 미삭제 조직·역할 조회, 인력등록·개인정보 갱신 인터페이스와 트랜잭션 테스트 완료 |
| `SYS-001` 프런트엔드 | 목록·옵션·등록 API 및 OpenAPI가 `READY`이고 개발 DB 통합 테스트 성공 |
| `RES-001` 전체 | 직원 최소 등록과 사용자 연결 정책 검증 완료 |

## 7. 구현 원칙

- 각 화면은 화면설계서, API 계약, 인수 시나리오와 구현 작업카드를 `READY`로 만든
  뒤 구현한다.
- 시스템 기준정보 삭제는 참조 데이터를 확인하고 논리삭제한다.
- 메뉴·역할 변경은 현재 세션을 유지하고 변경된 권한은 다음 로그인부터 적용한다.
- 인력과 사용자 개인정보 변경은 인력 모듈 공개 인터페이스를 통해 같은
  트랜잭션으로 동기화한다.
- 단계별 백엔드·프런트엔드 자동 테스트와 PostgreSQL 통합 검증을 통과한 뒤 다음
  단계로 이동한다.
